## §0 — THREE MAPS + WHY CYCLE-BASED

### §0.1 Three Core Maps v8.0

The framework describes the body-brain system from 2 perspectives, each as a separate map:

| # | Map | Perspective | For | File |
|---|---|---|---|---|
| 1 | Hardware Map | WHAT IS WHERE | Neuroscience researcher | Core-Hardware.md |
| 2 | **Software Map** | **HOW IT RUNS** | **Framework researcher** | **This file** |

Both maps describe the SAME system from different angles. Each can be read independently.
Like a computer: circuit diagram (Hardware) / code architecture (Software).
Framework interface: Ask-AI.md — AI reads the framework and adapts to each person's level of understanding.

### §0.2 Why v8.0 is Cycle-Based

Core-v7.5-Draft.md (2026-03) was organized as: 4 layers + 3 drives + Schema System.
That scaffolding enabled 35,000+ lines of analysis drilling (130+ files, 500+ citations).

After drilling: all mechanism files were compatible — layers/drives/schemas = observation labels,
not architecture. Cycle-based matches how the body-brain actually operates.

| V7.5 | V7.8 | Reason |
|---|---|---|
| 4 layers (L0-L3) | L0 circuit breaker + L1 inputs | L2 removed, L3 collapsed |
| 3 drives | Observation parameters | Drives = emergent, not operators |
| Schema System | Chunk-System (sole substrate) | Schema = observation label |
| Layer priority | Signal strength | Loudest wins, not highest layer |
| Cross-cutting | Properties of cycle | Cortisol = amplifier, Empathy = function |
| Feeling (implicit) | Explicit bridge | PFC observation interface |
| N separate modules | 1 Simulation-Engine | Shared substrate for Self-Pattern-Modeling, Imagine-Final, ... |
| Type A/Type B | Compiled/Fresh | Real processing spectrum, not content types |

### §0.3 Design Principles

1. **Follow mechanism, not organization**: architecture matches how the body-brain actually works
2. **Chunk-centric**: chunks = sole substrate, everything else = function or property
3. **Cycle, not layers**: continuous perception-action loop
4. **Observation parameters, not components**: Schema, Drives, Valence = ways of observing
5. **Development = chunk density**: same architecture, different accumulated chunks
6. **1 Engine, N Applications**: Self-Pattern-Modeling, Imagine-Final, Self-Observation = applications on the same Simulation-Engine
7. **Honest about limits**: descriptive architecture + hypothesis generator, not clinical protocol

### §1.4 Why Cycle, Not Layers

```
V7.5 LAYERS — what the problem was:

  L0 > L1 > L2 > L3 implies:
  ① Fixed priority ordering (L0 always beats L1)
  ② Drives = separate operators sitting above the substrate
  ③ Connection, meaning = had to be "slotted into" some layer
  ④ Development = unclear (which layer develops first?)

  But:
  ① Priority = signal strength, not layer rank
     → Strong pain beats mild hunger, but extreme hunger beats mild pain
     → = Signal strength decides, not layer position
  ② Drives = emergent patterns in the cycle, not operators
  ③ Connection = L1 multi-input aggregate + Entity-Compiled → solved
  ④ Development = chunk density → clear

V7.8 CYCLE — resolution:

  ① No fixed priority — signal strength decides (loudest wins)
  ② No separate operators — drives = observation labels
  ③ No placement problem — everything = function or property of cycle
  ④ Development = same cycle, more chunks → naturally explained

COMPARED TO 6-STEP LOOP (Body-Feedback.md §3):

  Body-Feedback.md §3 describes: Domain → Body sensory → Feedback → Schema Update → Action → Domain
  V7.8 REFINES: adds Feeling bridge + PFC observation + Orchestrate detail.
  = V7.8 = 6-step + explicit Feeling + PFC + Orchestrate.
```

## §6 — PFC (Observer + Orchestrator)

### §6.1 PFC = Observer, NOT Controller

```
🟡🟢 PFC CAN:
  → OBSERVE: read feeling, detect chunk activations
  → HOLD: ~4±1 WM slots (🟢 Cowan 2001) — amplify target pattern
  → SUPPRESS: block unwanted pattern (costly, unsustainable)
  → SEARCH: hold chunks → bias spreading activation direction
  → TYPE 4 LINKING: deliberate "thinking" (PFC hold A + B → check overlap → body vote)
  → SIMULATION-ENGINE: run applications (Self-Pattern-Modeling, Imagine-Final, Self-Observation)

🟡🟢 PFC CANNOT:
  → FEEL directly (must RECEIVE from body via feeling bridge)
  → COMPILE automatically (unconscious compiles, PFC only observes results)
  → RUN 95% background processing
  → OVERRIDE body-base when signal too strong (extreme hunger → eating frantically,
    meeting a crush → stuttering)
  → CONTROL motor directly (PFC holds "write dog" → unconscious executes keystrokes)

  ⭐ PFC HARDWARE SPECS (COMT, DRD4, WM capacity, modality balance,
```

### §12.4 Number Convention

```
🟡 NUMBERS IN THE FRAMEWORK = CALIBRATION ANCHORS:

  The framework uses numbers to help estimate DIRECTION + SCALE.
  NOT precise measurements. NOT fixed ratios.

  CONVENTION:
    ~X%     = order of magnitude ("roughly this scale")
    X-Y%    = illustrative range ("high variation, falls within this range")
    🟢 + %  = research-backed (has citation, empirical basis)
    🟡 + %  = framework estimate (calibration anchor, no measurement basis)

  EXAMPLE:
    "~95% unconscious / ~5% PFC"
      → INSIGHT: most processing does NOT need PFC.
      → NOT: PFC precisely runs 5.0% of the time.

    "7-layer fidelity: 100% → ... → 20-70%"
      → INSIGHT: fidelity DECREASES PROGRESSIVELY. Critical drop at labeling + explanation.
      → NOT: Feel-Observation is always exactly 85% accurate for everyone.

    "~90% Experience Compile / ~5% Expertise Compile"
      → INSIGHT: MOST behavior = direct compiled. Expertise = small portion.
      → NOT: exactly 90.0% of behavior is Experience Compile.

  → Reading numbers in the framework = reading DIRECTION + SCALE, not measurements.
```
