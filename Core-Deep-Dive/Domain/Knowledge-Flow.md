## 1. Flow — Internal → Output → External Chunk

```
⭐ DÒNG CHẢY CỐT LÕI — 1 vòng lặp xuyên suốt lịch sử:

  PERSON A (internal):
    → Chunks build bên trong (learn, trial-error, compile)
    → PFC draft + body check + domain test
    → = QUÁ TRÌNH tốn kém: thời gian, năng lượng, cortisol
    → = "XƯỞNG" — lộn xộn, phức tạp, cồng kềnh

  OUTPUT (tách khỏi person A):
    → Kết quả TÁCH ra: sản phẩm, công thức, kiến thức, công nghệ
    → Output TỒN TẠI RIÊNG — không cần person A duy trì
    → Person A có thể QUÊN chi tiết → output VẪN CÒN
    → = "BẢN THIẾT KẾ" — gọn, rõ, dùng được

  EXTERNAL CHUNK (người khác tiếp nhận):
    → Person B KHÔNG cần lặp lại quá trình của A
    → Person B chỉ cần LEARN CÁCH DÙNG output
    → Output của A trở thành 1 chunk RẺ cho B
    → = "NHÀ MÁY" — dùng thiết kế có sẵn, sản xuất, mở rộng

  PERSON B (internal, baseline CAO hơn):
    → B bắt đầu từ output của A (không từ zero)
    → B build TRÊN output A → tạo output MỚI → tách ra → ...
    → = Vòng lặp TIẾP TỤC: B → output B → external chunk cho C → ...

  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ PERSON A │         │ PERSON B │         │ PERSON C │
  │          │  output │          │  output │          │
  │ Internal │───────→ │ Internal │───────→ │ Internal │───→ ...
  │ (tốn kém)│  (rẻ)  │(baseline↑)│  (rẻ)  │(baseline↑↑)│
  └──────────┘         └──────────┘         └──────────┘
       ↑                     ↑                     ↑
    Xưởng               Build trên A          Build trên A+B


  VÍ DỤ — lửa:
    Người đầu tiên: hàng trăm lần thử → tìm ra cách mài gỗ (internal CỰC TỐN)
    Output: "mài gỗ = lửa" (kiến thức tách ra)
    Người sau: LEARN "mài gỗ" (external chunk — rẻ hơn NHIỀU so với tự tìm)
    → Người sau lại tìm ra đá lửa → output → external chunk → ...

  VÍ DỤ — đầu bếp:
    Đầu bếp A: 10 năm thử nghiệm → tìm ra công thức phở đặc biệt (internal tốn kém)
    Output: sách dạy nấu (tách ra, compressed)
    Đầu bếp B: đọc sách A → nấu ĐƯỢC phở đặc biệt trong vài tuần (external chunk rẻ)
    → B cải tiến thêm → output mới → đầu bếp C dùng → ...
    → Output B lại là base cho C

  → = "Xưởng VỨT → bản thiết kế GIỮ → nhà máy DÙNG → tới khi thiết kế MỚI tốt hơn"
  → = Flow KHÔNG BAO GIỜ DỪNG — mỗi output = baseline mới cho vòng sau

  ⭐ TẠI SAO FLOW XẢY RA — body-level drive behind sharing:
    → §1 mô tả CÁCH flow hoạt động (A→output→B)
    → Nhưng TẠI SAO A muốn share output cho B?
    → Empathy-Mirror.md §6.5: empathy-mirror = cơ chế phân phối tài nguyên TỰ NHIÊN
      → A đã compile (reward CỦA A ≈ 0 từ item đã biết)
      → A thấy B CHƯA biết → mirror deficit → A share → B improve → mirror reward cho A
      → = Body A TỰ DRIVE share vì mirror reward > giữ reward
      → = Flow xảy ra vì BIOCHEMISTRY, không chỉ vì "muốn giúp" hay "đạo đức"
    → Thầy giáo dạy, đầu bếp viết sách, developer open-source → CÙNG mechanism
```

```

    → = Triệu chunks internal: visual-spatial, physics, philosophy, toán
    → = QUÁ TRÌNH không ai lặp lại được (unique melody × unique path)

  OUTPUT = NÉN LẠI (vài ký tự, vài trang):
    → E=mc² = VÀI KÝ TỰ chứa decades suy nghĩ

    → = COMPRESSION RATIO cực cao

    → Sinh viên: học E=mc² trong vài giờ

     Compression càng cao → người sau learn càng rẻ → tiến càng nhanh.


  VÍ DỤ COMPRESSION SPECTRUM — cùng domain "di chuyển A → B":

    ┌───────────────────┬──────────────────┬──────────────────────────┐

    │ Máy bay           │ ngành hàng không  │ MUA VÉ (vài phút)       │

    → MỖI BƯỚC: internal cost TĂNG (công nghệ phức tạp hơn)
    → MỖI BƯỚC: external chunk cost GIẢM (người dùng cần biết ÍT hơn)
    → = Compression ratio TĂNG MÃI
    → = "Tiến bộ" = KHÔNG phải con người biết NHIỀU hơn
    → = "Tiến bộ" = output compressed TỐT hơn → dùng RẺ hơn → ai cũng dùng được

    → Người dùng máy bay KHÔNG CẦN biết khí động học
    → = External chunk compressed TỐT → user cost GẦN ZERO


  ⚠️ COMPRESSION KHÔNG PHẢI LÀ "ĐƠN GIẢN HÓA":
    → E=mc² KHÔNG đơn giản hóa vật lý → nó NÉN decades suy nghĩ vào vài ký tự
    → Xe máy KHÔNG đơn giản hóa di chuyển → nó NÉN engineering vào 1 sản phẩm
    → = Output CHỨA TOÀN BỘ complexity → nhưng user KHÔNG CẦN thấy complexity đó
    → = "Nén" ≠ "bớt" → "nén" = "giấu complexity bên trong, lộ interface bên ngoài"
```

## 4. Optimization Override — Cơ Chế Thay Thế Schema Cũ

```
⭐ Chunk {mài gỗ lấy lửa} KHÔNG bị vứt vì "cũ quá".
⭐ Chunk {mài gỗ lấy lửa} bị vứt vì chunk {bật lửa} TỐI ƯU HƠN và OVERRIDE.

  CƠ CHẾ:
    → Schema cũ VẪN ĐÚNG, vẫn HOẠT ĐỘNG (mài gỗ VẪN ra lửa)
    → NHƯNG: schema mới xuất hiện, tối ưu hơn (bật lửa: nhanh, rẻ, ai cũng dùng)
    → Mọi người TỰ CHỌN schema mới (vì tối ưu hơn)
    → Không ai "quyết định" vứt schema cũ → nó TỰ mất vì không ai duy trì
    → = LEGACY OBSOLETED: không sai, chỉ kém hơn → không ai dùng nữa

  PHÂN BIỆT 2 cơ chế "mất chunks":

    CÁ NHÂN — chunks decay (ref: Chunk.md §2):
      → Cơ chế: không dùng → retrieval path yếu → quên
      → Mục đích: freed PFC workspace cho chunks MỚI
      → = Não dọn nhà: vứt đồ không dùng → có chỗ cho đồ mới
      → = INTERNAL process

    TOÀN CẦU — optimization override:
      → Cơ chế: schema tối ưu hơn XUẤT HIỆN → mọi người TỰ CHỌN
      → Mục đích: toàn cầu map domain hiệu quả hơn
      → = Thị trường tự nhiên: sản phẩm tốt hơn → sản phẩm cũ mất
      → ⚠️ KHÔNG ai "xóa" schema cũ → nó tự mất vì không ai duy trì
      → = EXTERNAL process (selection giữa các output)

    → 2 cơ chế CÙNG chiều: cá nhân freed resource + toàn cầu freed schema space
    → = Cùng áp lực tối ưu: min cost × max coverage — ở MỌI level


  VÍ DỤ CHUỖI OVERRIDE (cùng domain "tạo nhiệt"):

    Mài gỗ → đá lửa → diêm → bật lửa → bếp ga → bếp từ
    ────────────────────────────────────────────────────────
    Mỗi bước: CÙNG domain → KHÁC schema → CHI PHÍ GIẢM
    Mỗi bước: schema cũ = legacy obsoleted (không sai, chỉ KÉM hơn)
    Mỗi bước: chunks cũ BIẾN MẤT khỏi flow (không ai teach nữa)
    Mỗi bước: output MỚI compressed hơn → dùng RẺ hơn


  → = "Tiến bộ" = KHÔNG phải biết NHIỀU hơn
  → = "Tiến bộ" = map domain HIỆU QUẢ hơn (ít chunks, nhiều coverage)
  → = Paradox: biết ÍT chi tiết hơn tổ tiên (không biết mài gỗ)
     nhưng map domain TỐT hơn tổ tiên (có lửa nhanh hơn, dễ hơn, rẻ hơn)


  ⚠️ OVERRIDE KHÔNG PHẢI LUÔN TỐI ƯU:
    → Đôi khi schema cũ BỊ MẤT dù CHƯA có thay thế hoàn toàn
    → VD: kỹ năng thủ công biến mất TRƯỚC khi máy móc thay thế hoàn toàn
    → VD: thuốc cổ truyền bị bỏ TRƯỚC khi y học hiện đại cover hết
    → = "Khoảng trống override": schema cũ mất, schema mới CHƯA đủ
    → = Trade-off: tốc độ override vs risk mất kiến thức
```

## 5. Dissonance → Output → Smooth — Chu Kỳ Mỗi Lần Flow

```
🟡 MỖI LẦN flow xảy ra = 1 chu kỳ 3 phase:

  PHASE 1 — DISSONANCE (đang tìm):
    → Chunks lộn xộn, xung đột, thử sai
    → Cortisol cao, PFC draft nhiều lần, body check liên tục
    → = "Giai đoạn R&D" — tốn kém, không thoải mái
    → = Xưởng chế tạo đang chạy: ồn ào, phế liệu, thất bại nhiều

  PHASE 2 — OUTPUT (tìm được):
    → Chunks kết tinh thành sản phẩm / kiến thức / công nghệ
    → Output TÁCH khỏi quá trình lộn xộn
    → = "Sản phẩm ra lò" — gọn, rõ, dùng được

  PHASE 3 — SMOOTH (ổn định):
    → Chunks công cụ lộn xộn CÓ THỂ LOẠI BỎ
    → Chỉ giữ output + kinh nghiệm (compiled sâu)
    → Melody trở về mượt mà
    → = "R&D xong → xưởng dọn sạch → dùng sản phẩm → thoải mái"

  CHU KỲ NÀY LẶP LẠI Ở MỌI SCALE:

    Micro (1 skill):
      Học đếm (dissonance: ngón tay, que tính, nhầm)
      → Biết đếm (output: skill compiled)
      → Quên que tính (smooth: chỉ giữ kết quả)

    Project (1 sản phẩm):
      Làm game 6 tháng (dissonance: code, bug, design thay đổi)
      → Game publish (output: sản phẩm tách ra)
      → Quên code chi tiết (smooth: chỉ giữ kinh nghiệm)

    Toàn cầu (1 breakthrough):
      Electricity R&D (dissonance: hàng trăm nhà phát minh thử sai)
      → AC/DC output (Tesla/Edison ra sản phẩm stable)
      → Thế giới DÙNG điện (smooth: không ai cần biết quá trình thử nghiệm)

    → = CÙNG pattern: dissonance → output → smooth
    → = CÙNG cơ chế: chunks lộn xộn → kết tinh → loại bỏ chunks công cụ
```
