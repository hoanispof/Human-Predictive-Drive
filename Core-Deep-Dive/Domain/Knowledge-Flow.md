## §1 — Flow — Internal → Output → External Chunk

```
⭐ THE CORE FLOW — 1 loop running throughout all of history:


  PERSON A (internal):
    → Chunks build internally (learning, trial-and-error, compilation)
    → PFC drafts + body checks + domain tests
    → = A COSTLY PROCESS: time, energy, sustained cortisol
    → = "WORKSHOP" — messy, complex, unwieldy


  OUTPUT (separated from Person A):
    → Result EXTRACTED: product, formula, knowledge, technology
    → Output EXISTS INDEPENDENTLY — does not require Person A to maintain it
    → Person A may FORGET the details → output REMAINS
    → = "BLUEPRINT" — concise, clear, usable


  EXTERNAL CHUNK (received by others):
    → Person B does NOT need to repeat A's process
    → Person B only needs to LEARN HOW TO USE the output
    → A's output becomes a CHEAP chunk for B
    → = "FACTORY" — uses the existing blueprint, produces, scales


  PERSON B (internal, HIGHER baseline):
    → B starts from A's output (not from zero)
    → B builds ON TOP OF A's output → creates NEW output → extracts it → ...
    → = Loop CONTINUES: B → output B → external chunk for C → ...


  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ PERSON A │         │ PERSON B │         │ PERSON C │
  │          │  output │          │  output │          │
  │ Internal │───────→ │ Internal │───────→ │ Internal │───→ ...
  │ (costly) │  (cheap)│(baseline↑)│ (cheap)│(baseline↑↑)│
  └──────────┘         └──────────┘         └──────────┘
       ↑                     ↑                     ↑
   Workshop            Builds on A           Builds on A+B


  EXAMPLE — fire:
    First person: hundreds of attempts → discovered wood-friction (EXTREMELY costly)
    Output: "friction = fire" (knowledge extracted)
    Later person: LEARNS "wood friction" (external chunk — MUCH cheaper than discovering)
    → Later person discovers flint → output → external chunk → ...

  EXAMPLE — chef:
    Chef A: 10 years of experimentation → developed a special pho recipe (costly internal)
    Output: a cookbook (extracted, compressed)
    Chef B: reads A's book → can cook the special pho within weeks (cheap external chunk)
    → B improves further → new output → Chef C uses it → ...
    → B's output becomes the base for C

  → = "Workshop DISCARDED → Blueprint KEPT → Factory USES → until a BETTER blueprint appears"
  → = Flow NEVER STOPS — each output = new baseline for the next round


  ⭐ WHY FLOW HAPPENS — body-level drive behind sharing:
    → §1 describes HOW flow works (A→output→B)
    → But WHY does A want to share with B?
    → Empathy-Mirror.md §6.5: the empathy-mirror = a NATURAL resource distribution mechanism
      → A has compiled (reward for A ≈ 0 from what A already knows)
      → A sees B has NOT yet learned → mirror deficit → A shares → B improves →
        mirror reward for A
      → = A's BODY DRIVES sharing because mirror reward > holding reward
      → = Flow happens because of BIOCHEMISTRY, not just "wanting to help" or "ethics"
    → Teacher teaching, chef writing a cookbook, developer releasing open-source → SAME mechanism
```

## §2 — Compression — Why Output Is Cheaper Than Internal

```
⭐ COMPRESSION = the core mechanism of flow:


  INTERNAL = COMPLEX (millions of chunks, many years):
    → Einstein: decades of thinking, thought experiments, trial-and-error
    → = Millions of internal chunks: visual-spatial, physics, philosophy, mathematics
    → = A PROCESS nobody can replicate (unique melody × unique path)

  OUTPUT = COMPRESSED (a few characters, a few pages):
    → E=mc² = A FEW CHARACTERS containing decades of thought
    → The paper: a few dozen pages of explanation
    → = EXTREME COMPRESSION RATIO

  EXTERNAL CHUNK = USABLE WITHOUT KNOWING THE PROCESS:
    → Student: learns E=mc² in a few hours
    → Does NOT need to think like Einstein
    → Does NOT need to know Einstein's internal path
    → Only needs to: understand the formula + how to use it
    → = Learn cost: a few hours vs decades → THOUSANDS-fold compression


  🟡 INSIGHT: compression level DETERMINES the rate of global progress.
     Higher compression → cheaper learning for those who follow → faster advance.


  COMPRESSION SPECTRUM EXAMPLE — same domain "moving from A to B":

  ┌───────────────────┬──────────────────┬──────────────────────────┐
  │ Schema            │ Internal cost    │ External chunk cost      │
  │                   │ (creator)        │ (user)                   │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Walking           │ evolution        │ 0 (body knows it already)│
  │                   │ (millions of     │                          │
  │                   │ years of bipedal │                          │
  │                   │ training)        │                          │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Horse riding      │ hundreds of      │ a few months to learn    │
  │                   │ years of         │                          │
  │                   │ trial-and-error  │                          │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Bicycle           │ decades of       │ a few days to learn      │
  │                   │ invention        │                          │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Motorcycle        │ decades of       │ a few weeks to learn     │
  │                   │ invention +      │ + purchase (money)       │
  │                   │ infrastructure   │                          │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Car               │ 100+ years of    │ a few months to learn    │
  │                   │ industry         │ + purchase               │
  ├───────────────────┼──────────────────┼──────────────────────────┤
  │ Airplane          │ aviation industry│ BUY A TICKET (minutes)   │

  │                   │                  │ TO FLY                   │

  │ Ride-hailing app  │ years of         │ TAP BUTTON (seconds)     │
  │                   │ development      │ NO NEED TO KNOW HOW      │
  │                   │                  │ TO DRIVE                 │
  └───────────────────┴──────────────────┴──────────────────────────┘

  → AT EACH STEP: internal cost INCREASES (technology grows more complex)
  → AT EACH STEP: external chunk cost DECREASES (users need to know LESS)
  → = Compression ratio KEEPS RISING
  → = "Progress" is NOT that humans know MORE
  → = "Progress" is that output is better compressed → cheaper to use → anyone can use it


  CITY A → CITY B — same domain, different compression:
    Walking: uses INTERNAL chunks (stamina, navigation) → months
    Motorcycle: uses EXTERNAL chunk (buy + ride) → 2 days
    Airplane: uses EXTERNAL chunk (buy ticket) → 2 hours
    → Each step: domain UNCHANGED (A→B), schema is better COMPRESSED
    → Airplane passenger does NOT need to know aerodynamics
    → = External chunk well-compressed → user cost approaches ZERO


  ⚠️ COMPRESSION IS NOT "SIMPLIFICATION":
    → E=mc² does NOT simplify physics — it COMPRESSES decades of thinking into a few characters
    → A motorcycle does NOT simplify travel — it COMPRESSES engineering into one product
    → = Output CONTAINS ALL THE COMPLEXITY → but users DO NOT NEED to see that complexity
    → = "Compress" ≠ "reduce" → "compress" = "hide complexity inside, expose interface outside"
```

## §3 — Domain Invariant — Schema Optimizes

```
⭐ INSIGHT: Domain NEVER CHANGES — only SCHEMA + KNOWLEDGE change.


  ⭐ META-FRAME — Vertical Loop (ref: Collective-Purpose.md §1):
    This insight = the FOUNDATION for the cosmic loop:
      Domain (fixed) → Body → Schema → Knowledge → Domain (more deeply mapped)
    THIS FILE describes HORIZONTAL flow (between individuals, across generations).
    Collective-Purpose.md describes the VERTICAL loop (humanity ↔ domain).
    2 loops nested: horizontal IS THE MEANS, vertical IS THE END (purpose framing).


  DOMAIN = CONSTANT (ref: Domain.md):
    → Humans NEED fire (domain: "generate heat") = UNCHANGED from millions of years ago
    → Humans NEED to move (domain: "A → B") = UNCHANGED
    → Humans NEED communication (domain: "transfer information") = UNCHANGED
    → Humans NEED shelter (domain: "protect the body") = UNCHANGED
    → = Body-need → creates domain-need → domain-need is INVARIANT
    → = Domain reality (physics, chemistry, society) is also INVARIANT
    → = What changes is NOT the domain — but HOW humans MAP onto the domain
    → = "Newly discovered domain" is actually "newly mapped territory"
      (Collective-Purpose.md §1: principles of AI existed since the Big Bang,
      humanity is only now mapping them)


  SCHEMA = VARIABLE (how body-need maps onto domain):
    → Fire: wood friction → flint → lighter → gas stove → induction cooker
    → Travel: walking → horse → car → airplane
    → Communication: shouting → letter → telephone → internet → AI
    → = SAME domain, SAME body-need → DIFFERENT schema every era


  OPTIMIZATION PRESSURE — always present, never stops:

    ① LOWEST COST: fewer chunks, less energy, less time → same output
       → "Wood friction" = 30 minutes + skill + effort → fire
       → "Lighter" = 1 second + 0 skill → fire
       → = SAME output (fire) → cost DROPS DRAMATICALLY

    ② BROADEST COVERAGE: 1 schema maps multiple domains
       → "Internet" = text + images + video + calls + commerce + ...
       → = 1 schema REPLACES: letters + TV + telephone + shops + ...

    → = Dual pressure: MIN cost × MAX coverage
    → = Schema achieving both → SPREADS NATURALLY → old schema FADES AWAY

  → Compression (§2) is the MANIFESTATION of this optimization pressure:
    → Better compressed output = LOWER cost for users
    → Output covering more domains = HIGHER coverage
    → = Global flow always moves toward: better compressed + broader coverage
```

## §4 — Optimization Override — How Old Schemas Are Replaced

```
⭐ The chunk {wood friction for fire} is NOT discarded because it is "old."
⭐ The chunk {wood friction for fire} is discarded because {lighter} is MORE OPTIMAL
   and OVERRIDES it.


  THE MECHANISM:
    → Old schema still CORRECT, still WORKS (wood friction still makes fire)
    → BUT: new schema appears, more optimal (lighter: fast, cheap, anyone can use it)
    → Everyone NATURALLY CHOOSES the new schema (because it is more optimal)
    → Nobody "decides" to discard the old schema → it FADES because nobody maintains it
    → = LEGACY OBSOLETED: not wrong, just INFERIOR → nobody uses it anymore


  DISTINGUISHING 2 mechanisms of "losing chunks":

    INDIVIDUAL — chunk decay (ref: Chunk.md §2):
      → Mechanism: not used → retrieval path weakens → forgotten
      → Purpose: frees PFC workspace for NEW chunks
      → = Brain tidying up: discards unused items → makes room for new ones
      → = INTERNAL process

    GLOBAL — optimization override:
      → Mechanism: more optimal schema APPEARS → everyone CHOOSES it naturally
      → Purpose: world maps domain more efficiently
      → = Natural market: better product → old product loses customers
      → ⚠️ NOBODY "deletes" the old schema → it fades because nobody maintains it
      → = EXTERNAL process (selection among outputs)

    → 2 mechanisms ALIGNED: individual frees resources + global frees schema space
    → = Same optimization pressure: min cost × max coverage — at EVERY level


  OVERRIDE CHAIN EXAMPLE (same domain "generating heat"):

    Wood friction → flint → matches → lighter → gas stove → induction cooker
    ──────────────────────────────────────────────────────────────────────────
    Each step: SAME domain → DIFFERENT schema → COST DECREASES
    Each step: old schema = legacy obsoleted (not wrong, just INFERIOR)
    Each step: old chunks DISAPPEAR from flow (nobody teaches it anymore)
    Each step: new output is better compressed → cheaper to use


  → = "Progress" is NOT knowing MORE
  → = "Progress" is mapping domain MORE EFFICIENTLY (fewer chunks, more coverage)
  → = Paradox: we know FEWER details than ancestors (don't know wood friction)
     but map domain BETTER than ancestors (have fire faster, easier, cheaper)


  ⚠️ OVERRIDE IS NOT ALWAYS OPTIMAL:
    → Sometimes the old schema is LOST even without a complete replacement
    → E.g.: craft skills vanish BEFORE machines fully replace them
    → E.g.: traditional medicine discarded BEFORE modern medicine covers everything
    → = "Override gap": old schema gone, new schema NOT YET sufficient
    → = Trade-off: speed of override vs risk of losing knowledge
```

## §5 — Dissonance → Output → Smooth — The Cycle of Each Flow Event

```
🟡 EACH FLOW EVENT = 1 cycle of 3 phases:


  PHASE 1 — DISSONANCE (searching):
    → Chunks are messy, conflicting, trial-and-error in progress
    → Cortisol high, PFC drafting repeatedly, body checking constantly
    → = "R&D phase" — costly, uncomfortable
    → = The workshop running: loud, producing scrap, many failures

  PHASE 2 — OUTPUT (found):
    → Chunks crystallize into a product / knowledge / technology
    → Output SEPARATES from the messy process
    → = "Product out of the oven" — concise, clear, usable

  PHASE 3 — SMOOTH (stable):
    → The messy scaffolding chunks CAN BE RELEASED
    → Keep only the output + compiled experience (deeply compiled)
    → Melody returns to smooth
    → = "R&D complete → workshop cleaned → using the product → comfortable"


  THIS CYCLE REPEATS AT EVERY SCALE:

    Micro (1 skill):
      Learning to count (dissonance: fingers, tally sticks, making errors)
      → Knows how to count (output: skill compiled)
      → Forgets the tally sticks (smooth: only keeps the result)

    Project (1 product):
      Building a game for 6 months (dissonance: code, bugs, design changes)
      → Game published (output: product extracted)
      → Forgets code details (smooth: only keeps the experience)

    Global (1 breakthrough):
      Electricity R&D (dissonance: hundreds of inventors trying, failing)
      → AC/DC output (Tesla/Edison produce stable products)
      → World USES electricity (smooth: nobody needs to know the trial-and-error process)

    → = SAME pattern: dissonance → output → smooth
    → = SAME mechanism: messy chunks → crystallize → release scaffolding chunks
```
