## 1. Vòng Đời Chunks — Từ Cá Nhân Tới Vĩnh Viễn

```
⭐ MỖI CHUNK có "vòng đời":

  ① SINH: cá nhân LEARN chunk (từ experience, school, AI, người khác)
  ② DÙNG: cá nhân dùng chunk để Novelty, giải quyết vấn đề, tạo output
  ③ OUTPUT: sản phẩm / kiến thức / công nghệ TÁCH khỏi cá nhân
  ④ QUÊN: cá nhân QUÊN chunk chi tiết (không cần giữ nữa)
  ⑤ SỐNG: output VẪN TỒN TẠI (dù cá nhân quên hoặc chết)
  ⑥ BASELINE: output TRỞ THÀNH baseline cho thế hệ sau
  ⑦ THAY THẾ: thế hệ sau tìm schema TỐI ƯU HƠN → output cũ DẦN bị thay thế

  → Chunk = TẠM THỜI cho cá nhân
  → Output = CÓ THỂ VĨNH VIỄN cho tập thể (cho tới khi bị OVERRIDE)
  → = "Xưởng chế tạo VỨT → bản thiết kế GIỮ → nhà máy DÙNG → tới khi thiết kế MỚI tốt hơn"

DISSONANCE → OUTPUT → SMOOTH (chu kỳ toàn cầu):

  Khi ĐANG tìm schema mới: chunks lộn xộn = DISSONANCE MELODY
    → Thử sai, trial-error, cortisol, nhiều chunks conflict nhau
    → = "Giai đoạn R&D" — lộn xộn, tốn kém, không thoải mái

  Khi TÌM ĐƯỢC: chunks công cụ → kết tinh thành OUTPUT
    → Output = sản phẩm, công nghệ, phương pháp — TÁCH khỏi dissonance

  Khi output ĐÃ STABLE: chunks công cụ CÓ THỂ LOẠI BỎ → về SMOOTH MELODY
    → Không cần giữ quá trình lộn xộn → chỉ giữ kết quả
    → = "R&D xong → sản phẩm ra → xưởng dọn sạch → thoải mái"

  Ở level TOÀN CẦU: mỗi breakthrough = 1 chu kỳ dissonance → smooth:
    → Electricity: hàng trăm nhà phát minh thử sai (dissonance)
      → Tesla/Edison ra output (AC/DC) → output stable
      → Chunks "thử nghiệm điện" BIẾN MẤT → mọi người CHỈ DÙNG output
    → Internet: decades nghiên cứu lộn xộn (ARPANET, protocols, thử sai)
      → Output stable (TCP/IP, HTTP) → mọi người DÙNG mà không cần biết quá trình
    → = Dissonance = CÁI GIÁ → output = CÁI ĐƯỢC → smooth = TRẠNG THÁI SAU
```

## 2. Xưởng Chế Tạo × Nhà Máy — Metaphor Cốt Lõi

```
XƯỞNG CHẾ TẠO (cá nhân/nhóm Novelty = DISSONANCE phase):
  → CỰC TỐN: thời gian, năng lượng, cortisol, trial-error
  → CỰC CỒNG KỀNH: chunks phức tạp, PFC draft nhiều lần, body check
  → OUTPUT: 1 "bản thiết kế" (sản phẩm, công thức, phương pháp)
  → SAU KHI XONG: xưởng KHÔNG CẦN NỮA (dissonance → smooth)
    → Chunks chi tiết: DẦN QUÊN (body freed resource)
    → Chỉ GIỮ: kinh nghiệm + kiến trúc cơ bản (= "bản thiết kế" rút gọn)
  → = "100 người mở xưởng → 1 người ra thiết kế tốt nhất → 99 xưởng còn lại VỨT"

NHÀ MÁY SẢN XUẤT (tập thể dùng output = SMOOTH phase):
  → RẺ: dùng "bản thiết kế" có sẵn → KHÔNG cần novelty lại
  → HIỆU QUẢ: sản xuất HÀNG TRIỆU bản → mỗi bản CỰC RẺ
  → = "1 thiết kế → triệu sản phẩm → triệu người dùng"
  → Tới khi: schema MỚI tối ưu hơn → nhà máy CŨ đóng → nhà máy MỚI mở

  Ví dụ cụ thể:
    Tesla (xưởng): 20 năm Novelty → output: AC electricity
    GE/Westinghouse (nhà máy): dùng thiết kế Tesla → sản xuất → TOÀN THẾ GIỚI dùng AC
    Tesla chết → "xưởng" mất → nhưng "thiết kế AC" VẪN chạy
    → = Chunks Tesla QUÊN/MẤT → output Tesla VĨNH VIỄN (cho tới khi energy mới OVERRIDE)
```

## 4. Domain Bất Biến — Schema Tối Ưu

```
⭐ INSIGHT CỐT LÕI: Domain KHÔNG BAO GIỜ thay đổi — chỉ SCHEMA thay đổi.

  DOMAIN = HẰNG SỐ (ref: Architecture-v7.5 §7):
    → Con người CẦN lửa (domain: "tạo nhiệt") = KHÔNG ĐỔI từ triệu năm trước
    → Con người CẦN di chuyển (domain: "A → B") = KHÔNG ĐỔI
    → Con người CẦN communication (domain: "transfer info") = KHÔNG ĐỔI
    → Con người CẦN shelter (domain: "bảo vệ body") = KHÔNG ĐỔI
    → = Body-need → tạo ra domain-need → domain-need BẤT BIẾN
    → = Domain reality (vật lý, hóa học, xã hội) cũng BẤT BIẾN
    → = Cái thay đổi KHÔNG PHẢI domain — mà là CÁCH con người MAP vào domain

  SCHEMA = BIẾN SỐ (cách map body-need vào domain):
    → Lửa: mài gỗ → đá lửa → bật lửa → bếp ga → bếp từ
    → Di chuyển: đi bộ → ngựa → xe hơi → máy bay
    → Communication: hét → thư → điện thoại → internet → AI
    → = CÙNG domain, CÙNG body-need → KHÁC schema mỗi thời đại
    → = Schema = "cách chơi bài nhạc" trên "nhạc cụ" domain

  ÁP LỰC TỐI ƯU — luôn có, không bao giờ dừng:

    ① CHI PHÍ THẤP NHẤT: ít chunks, ít energy, ít thời gian → cùng output
       → "Mài gỗ" = 30 phút + skill + effort → lửa
       → "Bật lửa" = 1 giây + 0 skill → lửa
       → = CÙNG output (lửa) → chi phí GIẢM CỰC MẠNH

    ② COVER NHIỀU DOMAIN NHẤT: 1 schema map được nhiều domains
       → "Internet" = text + hình + video + call + mua bán + ...
       → = 1 schema THAY THẾ: thư + TV + điện thoại + cửa hàng + ...
       → = Cover NHIỀU domain-needs bằng 1 schema duy nhất

    → = Áp lực kép: MIN cost × MAX coverage
    → = Schema nào đạt cả 2 → TỰ NHIÊN lan rộng → schema cũ TỰ BIẾN MẤT

  CƠ CHẾ THAY THẾ — OPTIMIZATION OVERRIDE (KHÔNG phải time decay):

    ⭐ Chunk {mài gỗ lấy lửa} KHÔNG bị vứt vì "cũ quá"
    ⭐ Chunk {mài gỗ lấy lửa} bị vứt vì chunk {bật lửa} TỐI ƯU HƠN và OVERRIDE

    → Schema cũ VẪN ĐÚNG, vẫn HOẠT ĐỘNG (mài gỗ VẪN ra lửa)
    → NHƯNG: không ai DUY TRÌ nữa → legacy obsoleted
    → = Không ai "quyết định" vứt mài gỗ
    → = Mọi người TỰ CHỌN bật lửa (vì tối ưu hơn)
    → = Selection tự nhiên ở level SCHEMA: schema fit hơn → sống, schema kém → biến mất

    Phân biệt 2 cơ chế "mất chunks":
    ┌────────────────────────────────────────────────────────────┐
    │  CÁ NHÂN: chunks decay = NÃO freed resource               │
    │    → Cơ chế: không dùng → retrieval path yếu → quên       │
    │    → Mục đích: freed PFC workspace cho chunks MỚI          │
    │    → = Dọn nhà: vứt đồ không dùng → có chỗ cho đồ mới    │
    │                                                            │
    │  TOÀN CẦU: chunks override = SCHEMA MỚI thay schema cũ    │
    │    → Cơ chế: schema tối ưu hơn XUẤT HIỆN → mọi người CHỌN │
    │    → Mục đích: toàn cầu map domain hiệu quả hơn           │
    │    → = Thị trường: sản phẩm tốt hơn → sản phẩm cũ mất    │
    │    → ⚠️ KHÔNG ai "xóa" schema cũ — nó TỰ mất vì không    │
    │      ai duy trì                                            │
    └────────────────────────────────────────────────────────────┘

  VÍ DỤ CHUỖI OVERRIDE (cùng domain "tạo nhiệt"):

    Mài gỗ → đá lửa → diêm → bật lửa → bếp ga → bếp từ
    ────────────────────────────────────────────────────────
    Mỗi bước: CÙNG domain → KHÁC schema → CHI PHÍ GIẢM
    Mỗi bước: schema cũ = legacy obsoleted (không sai, chỉ KÉM hơn)
    Mỗi bước: chunks cũ BIẾN MẤT khỏi global melody (không ai teach nữa)
    Mỗi bước: smooth melody MỚI = dùng output mới mà KHÔNG cần biết cách cũ

  → = "Tiến bộ" = KHÔNG phải biết NHIỀU hơn
  → = "Tiến bộ" = map domain HIỆU QUẢ hơn (ít chunks, nhiều coverage)
  → = Paradox: biết ÍT chi tiết hơn tổ tiên (không biết mài gỗ)
     nhưng map domain TỐT hơn tổ tiên (có lửa nhanh hơn, dễ hơn, rẻ hơn)
```

## 5. Chunks "Vứt Đi" — Tại Sao ĐÚNG Thiết Kế

```
🟡 "Quên chunks cũ" = KHÔNG phải "lãng phí" → là TỐI ƯU:

  NÃO CÓ RESOURCE HỮU HẠN:
    → 86 tỷ neurons × 7000 connections = NHIỀU nhưng KHÔNG vô hạn
    → Giữ MỌI chunks = TRÀN → PFC workspace BỊ chiếm → Novelty KHÓ
    → QUÊN chunks không cần = FREED resource → PFC workspace MỞ → Novelty DỄ
    → = "Dọn nhà = vứt đồ cũ → CÓ CHỖ cho đồ mới"

  "QUÊN" ≠ "MẤT hoàn toàn" (ref: Chunk-And-PFC.md §2):
    → Chunks compiled SÂU: VẪN CÒN ở vô thức (chỉ PFC không access nữa)
    → Nếu CẦN: có thể RE-ACTIVATE (chậm, nhưng ĐƯỢC)
    → NHƯNG: chunks KHÔNG dùng lâu → DẦN yếu (use it or lose it)
    → = "Đồ trong kho LÂU quá → hư dần → nhưng khung VẪN CÒN"

  TẠI SAO "giữ kinh nghiệm, vứt chi tiết":
    → Kinh nghiệm = PATTERN chung (áp dụng được cho domain KHÁC)
    → Chi tiết = SPECIFIC cho 1 project (chỉ dùng cho ĐÚNG project đó)
    → = Kinh nghiệm = schema TỔNG QUÁT → cover nhiều domains → GIỮ
    → = Chi tiết = schema CỤ THỂ → cover 1 domain → VỨT
    → = Não tối ưu CÙNG áp lực với toàn cầu: min cost × max coverage
    → = Cá nhân và toàn cầu CÙNG optimization pattern (§4)
```

## 7. Giá Trị Cá Nhân vs Giá Trị Tập Thể

```
⭐ 2 LOẠI GIÁ TRỊ — KHÁC nhau hoàn toàn:

  GIÁ TRỊ CÁ NHÂN = body reward (CHỈ khi ĐANG SỐNG + ĐANG feel):
    → Novelty sướng, Connection ấm, Mastery proud = giá trị CÁ NHÂN lúc đó
    → Khi CHẾT: giá trị cá nhân = ZERO (body không còn → reward không còn)

  GIÁ TRỊ TẬP THỂ = output tồn tại (CÒN sau khi cá nhân MẤT):
    → Sản phẩm, con cái, ảnh hưởng, kiến thức → TỒN TẠI trong global melody
    → Khi CHẾT: giá trị tập thể CÓ THỂ VẪN CÒN (output vẫn sống)

  2 CÁI CÓ THỂ LỆCH:
    Tesla: giá trị tập thể ★★★★★ + giá trị cá nhân cuối đời ★☆☆☆☆
      → Output CỰC LỚN (AC) → NHƯNG cuối đời: cô đơn, body-base thiếu
    Người bình thường: giá trị tập thể ★★☆☆☆ + giá trị cá nhân ★★★★☆
      → Output NHỎ → NHƯNG body-base ok + connection ok → sướng

  Qua lens §4 (domain bất biến):
    → Giá trị tập thể = output MAP domain tốt hơn cho NHIỀU người
    → Giá trị cá nhân = body reward TRONG QUÁ TRÌNH map domain
    → 2 cái KHÔNG tự động đi kèm → CẦN chủ động balance
```
