> Human body-feedback is a **continuous cycle** where:

> - **Imagine-Final**: reference patterns at all scales providing "should be" expectations
> - **Anchor-Schema**: sync points keeping the whole system aligned
> - **Personal-Melody**: emergent property of the whole — how "resonant" the integrated state is
>
> The cycle iterates across timescales (milliseconds to decades). Body-feedback is continuous, the feeling layer develops over a lifetime, chunks accumulate + drift, and the melody evolves.

### §2.4 — Running example: "Wake up, see the kitchen, feel hungry"

Tracing a simple event through the loop:

**Step 1**: Open eyes → visual input (kitchen scene)
- Timing: 10–100ms perception

**Step 2**: Unconscious processes
- Chunks about kitchen fire: "breakfast location"
- Compiled schema: "morning routine"
- Imagine-Final unconscious: expected = "breakfast soon"
- Body-base check: blood glucose low (interoception)
- Cascade: "hunger chunks" → food chunks → specific food preferences
- Timing: 100–500ms

**Step 3**: Upstream integration
- Insula integrates: hunger signal from gut + visual food-context + glucose level
- Produces: "eat soon" coherent signal

**Step 4**: Body signal
- Mild dissonance: body-need (fuel) unmet
- Not high intensity (still baseline awake)
- Direction: toward food

**Step 5**: PFC gateway
- Attention reaches signal: "I'm hungry"
- Labels feeling: "hunger" (chunks compiled from lifetime)
- Simulates options: "make eggs" vs "cereal" vs "toast"
- Body pre-feels each option → "eggs sounds good"
- Chooses: make eggs

**Step 6**: Schema update minor
- No new compile (routine)
- Slight strengthening of "breakfast decision" schema possible

**Step 7**: Action
- Compiled: walk to fridge, get eggs, start cooking
- Mostly auto with occasional PFC check

**Step 8**: Action affects domain
- Eggs cooked → food on plate
- Eating → hunger signal decreases

**LOOP**: Back to Step 1 with new sensory input (taste of eggs, fullness emerging)

**→ Simple event traces cleanly through loop.** Complex events (learning, trauma, relationships) trace the same architecture but with much richer activation and longer timeframes.

### §2.5 — Who handles each step (unconscious vs PFC split)

| Step | Primary handler | PFC role |
|---|---|---|
| 1 Sensory input | Body sensors | None (unless directing attention) |
| 2 Unconscious processing | Unconscious (parallel, ~95%) | None directly |
| 3 Upstream integration | Subcortical (insula, ACC) | None |
| 4 Signal generation | Body-base | None |
| 5 PFC gateway | **PFC (~5%)** | Observe, label, simulate, choose |
| 6 Schema update | Unconscious (compile) + PFC (create new) | Create new chunks deliberately |
| 7 Action | Compiled (~80%) + PFC directed (~20%) | Direct novel action |
| 8 Domain effect | Body motor | Monitor |

**Critical**: PFC engages only at Steps 5–6–7, and only when needed. The rest is unconscious.

### §2.6 — Parallel cycles running simultaneously

🟡 **Framework claim**:

> The body runs **many interface loops simultaneously**. Unconscious processing is massively parallel. PFC is serial.

**Example — at any given moment**:
- Hunger loop (every few hours iteration)
- Social relationship loop (every interaction iteration)
- Temperature regulation loop (continuous micro-iteration)
- Learning project loop (days to weeks iteration)
- Career arc loop (years iteration)
- Health maintenance loop (continuous background)
- … hundreds more

Each loop operates on its own timescale. PFC attention allocates between them based on signal salience.

**Implication for drill**: Dissonance/reward cases typically involve a **specific loop** at a specific timescale. Mini dissonance (B1) = short-term loops. Trauma (B3) = long-term loop stuck. Hedonic trap (B4) = multiple long-term loops interacting.

## §3 — Imagine-Final Role in Cycle

### §3.1 — Imagine-Final as reference pattern

🟡 Framework (Imagine-Final.md §1 + §1.5):

> Imagine-Final = **reference pattern** against which actual state is compared. Appears at all scales (micro to lifelong) and at both levels (unconscious ~95% + PFC ~5%).

**Role in cycle**:
- Provides the "expected state" for Step 2 comparison
- Provides the "destination" for Step 5–7 action selection
- Provides the "meaning" for sustained effort through dissonance

Without Imagine-Final:
- No expected state → body doesn't know what's "off"
- No destination → no action direction
- No meaning → dissonance feels arbitrary

### §3.2 — 5-Phase Lifecycle integrated with cycle

🟡 Imagine-Final §1.5: **5 phases** — BUILD → SAVE → BACKGROUND → RELOAD → REFINE

**Mapping to interface loop**:

**Phase 1 BUILD**:
- Trigger: dissonance signal at Step 4 reaches PFC at Step 5
- PFC constructs Imagine-Final candidate
- Body pre-feels → confirms or rejects
- Selected candidate → Imagine-Final held in working memory

**Phase 2 SAVE**:
- PFC switches to other tasks (attention shifts)
- Imagine-Final moves to storage (hippocampus)
- Proto-schema state — not yet compiled deep

**Phase 3 BACKGROUND**:
- **Cycle continues** for other loops
- Unconscious processes chunks related to the saved Imagine-Final
- Spreading activation cascades connect chunks (P3 §7)
- **Happens automatically** regardless of PFC focus
- Sleep cycles consolidate significantly

**Phase 4 RELOAD**:
- External trigger or internal readiness
- PFC pulls Imagine-Final back to active workspace
- Often contains **new integrated context** from Phase 3
- Outcomes: unchanged / clearer / shifted

**Phase 5 REFINE**:
- PFC updates Imagine-Final based on new context
- Loops back to Phase 1 of the next iteration

### §3.3 — Dissonance and Imagine-Final interaction

🟡 Key framework insight (Imagine-Final §1 "dormant during harmony, fires during dissonance"):

**When harmony**:
- Body-base met → no dissonance signal
- Imagine-Final dormant (no need to check)
- PFC not engaged
- "Autopilot" state

**When dissonance**:
- Body-base signal fires
- Imagine-Final activates as reference for "what should be"
- PFC engages to compare and find a path
- Motivation emerges

**→ Imagine-Final is fire-on-demand.** Not continuously checking. Activates when body signals need.

### §3.4 — Why Imagine-Final clarity matters

**Clear Imagine-Final**:
- Body simulates vividly → strong opioid preview → strong motivation
- PFC has a clear target → directs attention well
- Chunks compile with clear "why"
- Dissonance along the way has meaning ("worth it for the goal")

**Fuzzy Imagine-Final**:
- Weak body simulation → weak motivation
- PFC without clear target → scattered attention
- Chunks compile without "why"
- Dissonance feels meaningless → burnout risk

**Framework prediction**: Activities with a clearer Imagine-Final sustain longer + produce more reward than vague ones.

### §3.5 — Positive feedback loop

🟡 Critical insight (Imagine-Final §1 "evolution" + positive feedback loop):

```
Clear Imagine-Final
      ↓
Strong motivation
      ↓
Build chunks (progress)
      ↓
Domain understanding grows
      ↓
Imagine-Final gains clarity + detail
      ↓
EVEN stronger motivation
      ↓
... repeats, compounding ...
```

**Einstein case study** (detailed in §6): Imagine-Final started with "Newtonian physics has contradictions." Over years, chunks accumulated. Imagine-Final grew: "there must be a unified framework." Growing clarity fed motivation. Successful work added chunks → even clearer. Compounding effect produced revolutionary insight.

**Hedonic trap case** (detailed in §7): Imagine-Final "wealth = happiness" is **fuzzy** about what happiness actually means. Progress toward wealth doesn't clarify what happiness requires. Imagine-Final doesn't sharpen. Positive feedback loop broken.

## §4 — Anchor-Schema Role in Cycle

### §4.1 — Anchor-Schema as sync point

🟡 Framework (Anchor-Schema.md §1 + §3):

> Anchor-Schema = **sync point** keeping the entire unconscious system aligned. The "main player" in the orchestra — others follow.
>
> Without an active anchor: system drifts, cacophony, no direction.
> With an active anchor: all schemas align toward it, coherent action emerges.

### §4.2 — Main-chord lens

Precise articulation:
> "Anchor-Schema is critically needed during strong dissonance, forcing a few patterns to fire clearly in the brain so that other schemas can look toward them and synchronize. If viewed through the melody lens — like the main chord needing more volume — then all other musicians easily know who to follow, and the whole orchestra synchronizes despite many players."

**Framework fully validates.** Anchor-Schema.md §1 uses the exact same metaphor (orchestra + main player).

### §4.3 — Role in cycle

**In normal operation**:
- Step 5 PFC draws from the active Anchor-Schema to decide action
- Step 7 Action direction provided by the anchor
- Anchor persists across cycles (not reset each iteration)

**When anchor absent**:
- Each loop runs independently
- No overall direction
- "Lost" / "adrift" feeling (Boredom-Analysis §1 Anchor-Schema lens)

### §4.4 — 4 Sources of Fill — which dominates per context

🟡 Anchor-Schema.md §3: 4 sources that fill Anchor-Schema:

1. **PFC Imagine-Final** (top-down, ~5% of decisions)
2. **Hippocampus simulation** (mid-level, ~10%)
3. **Compiled schemas** (bottom-up, ~80% dominant) ⭐
4. **External inject** (outside-in, variable)

**Implications**:
- Most of life ~80% auto-anchored by compiled schemas — no PFC needed
- Only novel/conflict situations require PFC Imagine-Final as source
- Culture + family inject external anchors (religion, career expectations, values)

### §4.5 — Trust binding — why anchors hold or collapse

🟡 Anchor-Schema §2:

> **Trust ≥ Cost → anchor holds. Trust < Cost → anchor collapses.**

**Cost** = time + energy + risk + opportunity cost + accumulated dissonance
**Trust** = confidence in reward + reliability of source + compiled deep evidence

**Positive vs Negative trust** (P2 §5.5 drilled):
- Positive trust → opioid reward sustainable
- Negative trust → relief reward diminishing

**Example calculation**:
- Father Promises Toy Car (P2 §5.3): Cost = 2 days of hard math work. Trust = positive (father reliable + clear reward) → holds easily
- Forced Punishment Learning (P2 §5.2): Cost = daily forever. Trust = negative (avoid punishment only) → unstable, collapses into chronic dissonance

### §4.6 — PFC serial processing — "Can you learn to drive and study math simultaneously?"

**Question**:
> "If learning to drive, while also studying new mathematics, can PFC hold both goals at once?"

🟡 **Framework answer**: **NO** (directly simultaneously).

**Reasons**:
- PFC = serial processor (Cowan 2001 🟢 working memory 4±1 items)
- 1 active focus at a time for "active" work (Imagine-Final §1.5 Foundation 1)
- Anchor-Schema needs **main player** volume high → can amplify only 1 at a time

**BUT alternate mode is possible**:
- Hold 2 Imagine-Finals in storage
- Switch between them (alternate days, alternate hours)
- Each iteration: reload → brief active work → save → switch
- Progress slower than single focus but achievable

**Framework prediction**:
- Both skills progress concurrently if alternated
- Each progresses ~50% as fast as single focus
- Switching cost reduces total throughput
- Some "unconscious practice" happens during the other's focus time (Phase 3 background for inactive Imagine-Final)

**When alternating works well**:
- Both activities can compile chunks (not PFC-only work)
- Rest/sleep sufficient for both to consolidate
- Neither activity actively conflicts with the other

**When alternating fails**:
- Activities compete for the same resources (both need the same chunks base)
- Exhaustion prevents both from progressing
- One becomes threatening (must finish) → takes over → other neglected

### §4.7 — Anchor-Schema explains why PFC is serial

🟡 **Elegant framework insight**:

> PFC serial processing constraint isn't just a "limitation." It's an **architectural feature** — it forces the system to have ONE main player so the orchestra can sync.

**If PFC could process in parallel**:
- Multiple equally-weighted anchors active
- Orchestra confused about which to follow
- Cacophony, no coherent action
- Evolutionary disadvantage (organism can't commit to a single response in an emergency)

**Serial constraint ensures**:
- One clear "winning anchor" at a time
- All schemas sync toward it
- Coherent behavior emerges
- Switching possible but not simultaneous

**→ Serial PFC is a FEATURE, not a bug** — required by the anchor sync architecture.

### §4.8 — Compiled anchors vs deliberate anchors

🟡 Anchor-Schema §3 ③ (compiled source ~80%) vs ① (PFC Imagine-Final ~5%):

**Compiled anchors**:
- Calling / vocation, expertise domain, deep values
- Auto-fire, don't need PFC maintenance
- Strong, stable, define "who you are"
- Examples: "I am an engineer," "I love my family," "I believe in science"

**Deliberate PFC anchors**:
- Novel decisions, conflicts, long-term planning
- Require active PFC engagement
- Weaker, more flexible
- Examples: "buy a house this year," "learn skill X," "invest in opportunity Y"

**Framework implication**: Life mostly runs on compiled anchors. PFC anchors used only for ~5% of decisions. Identity = compiled anchors network.

**Integration with dissonance/reward drill**:
- Most daily reward/dissonance comes from compiled-anchor alignment check
- Big life decisions (trauma loop, hedonic trap, C5 schema shift) require PFC deliberate anchor engagement
- **Trauma loop** often involves compiled anchors being damaged/degraded
- **Hedonic trap** often involves PFC anchor (money goal) neglecting compiled anchors (friendships, interests)

## §5 — Personal-Melody Evolution

### §5.1 — Personal-Melody as emergent property

🟡 Framework (Personal-Melody.md §1–§5):

> Personal-Melody = **emergent property** of integrated body-feedback state. It's what the whole architecture **sounds like** when running.
>
> Not a single schema. Not a single feeling. The coherent (or cacophonous) product of all schemas, chunks, signals, and body states integrated.

**Metaphor**: An orchestra produces music. The melody ≠ any single instrument. It emerges from all playing together.

**Applied**: A person's "how life feels" = their melody. Smooth = preconditions met across life domains. Rough = multiple dissonances active simultaneously.

### §5.2 — Melody smoothness determinants

🟡 Framework:

**Smooth melody**:
- Multiple body-needs met
- Imagine-Finals aligned (not conflicting with each other)
- Chunks base rich enough for cascade reward
- Anchor-Schemas stable + positive trust
- Sleep + nutrition + physical + social baseline adequate
- Minimal chronic dissonance

**Rough melody**:
- Multiple unmet needs
- Conflicting Imagine-Finals (Anchor-Schema collapse)
- Chunks base decaying (hedonic trap)
- Trust violations (negative anchors dominant)
- Physical wear (Cortisol-Baseline imbalance)
- Chronic dissonance accumulating

### §5.3 — Arcs and Valleys (Melody-Arc §2–§3)

🟡 Melody builds through arcs:

**Arc structure**:
- Start: dissonance (new chunks to accumulate)
- Middle: valley (biggest dissonance, "peak pain")
- End: integration (chunks fit, melody raised)

**Repeats across life**:
- Multi-year arcs (career build, relationship build, skill mastery)
- Each successful arc raises the baseline (melody higher)
- Failed arcs: abandoned → chunks wasted, slight melody degradation

**The Valley** (Melody-Arc §3.3):
- Middle of arc = most invested + least reward visible
- "Dark night" feeling
- Dropout point → failed arc
- Pushing through → breakthrough possible

**Framework prediction**: People's ability to handle valleys correlates with:
- Imagine-Final clarity (stronger → more tolerance)
- Positive trust (more → more persistence)
- Social support (external anchor reinforcement)
- Prior successful valley experience (meta-schema "valleys end")

### §5.4 — Baseline shift (hedonic adaptation)

🟢 Hedonic adaptation research (Brickman 1978 + extensive subsequent):

- Positive events → baseline shifts up
- Negative events → baseline shifts down
- Body adapts toward new baseline within 6–18 months typically
- Rewards requiring constant baseline-exceeding become unsustainable

**Framework integration**:
- VTA habituation (P3 §2.8) = the neural mechanism of hedonic adaptation
- Multiple domains adapt independently
- Some domains adapt faster (sensory), others slower (meaningful work)

**Implication for drill**: Hedonic trap (P2 §9) is partly hedonic adaptation + schema maintenance cost combined. Not just "adapted to more wealth" but "resources invested in the wealth domain while neglecting relationship/interest domains."

### §5.5 — Personal-Melody as reward source

🟡 Framework claim (extending Personal-Melody.md):

> When melody is smooth, a **baseline reward state** emerges — not "euphoric" but **contented, engaged, meaningful**. This is sustained reward (P3 §10).

**Smooth melody indicators**:
- Work feels engaging most days
- Relationships feel nourishing
- Body feels rested
- Minor frustrations don't cascade
- Sense of "life is good"

**Rough melody indicators**:
- Work feels draining
- Relationships feel conflicted
- Body chronically tired
- Minor frustrations cascade
- Sense of "life is hard"

**These are not personality traits or mood.** They are emergent melody quality from the architecture's current state.

### §5.6 — Melody evolution across a lifetime

🟡 Framework trajectory:

**Childhood** (age 0–12):
- Chunks base building rapidly
- Melody mostly determined by caregiver + environment
- Feeling layer developing (self-signal interoception scaffolding)
- High sensitivity to trauma (developmental window)

**Adolescence** (age 12–25):

> **Body signal (reward, dissonance, or neutral) emitted at Step 4 of the interface loop is a deterministic function of 5 preconditions evaluated at the moment of schema-body coherence check.**

### §9.2 — The 5 Preconditions (Final Definitions)

#### §9.2.1 — Precondition 1: Schema Pending Status

**Definition**:
> Body must have **active unresolved body-need** OR **schema seeking upgrade/resolution** at the time of input evaluation.

**Operationalization**:
- Body-need = signal from L1 body-inputs (Body-Input-Enumeration §4) indicating homeostatic gap OR

- Both forms count

**Failure mode**: "Don't need it / don't want it" — satiety, indifference, completion

#### §9.2.2 — Precondition 2: Chunks Base Adequacy

**Definition**:
> Body must have **sufficient compiled chunks** to decode the incoming pattern — i.e., pattern matches enough existing chunks to be recognized as meaningful.

**Operationalization**:
- Chunks base evaluated per-domain (art chunks for art input, math chunks for math input, etc.)

**Failure mode**: "Don't understand it / don't know what it is" — confusion, alien perception

#### §9.2.3 — Precondition 3: Prediction-Delta Threshold

**Definition**:

**Operationalization**:
- Habituation-based (prediction-delta, not "prediction error" — PFC-Analysis §8.3)

**Failure mode**: "Normal / used to it" — habituation, routine, no salience

**Test**: Daily positive routine events often fail to register as rewarding despite objective goodness

#### §9.2.4 — Precondition 4: Goldilocks Zone Position

**Definition**:

**Operationalization**:

**Failure modes**:
