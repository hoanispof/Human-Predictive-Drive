## §1 — THE CORE PROBLEM: "SPEAKING" ≠ "LISTENER UNDERSTANDING"

### §1.1 — "Not knowing = no gap" applied to communication

```
⭐⭐ FOUNDATIONAL PRINCIPLE (Gap-Direction.md §3):

  "Not knowing = no gap."
  You CANNOT miss something you do not know exists.
  A gap NEEDS surrounding chunks to EXIST.
  No surrounding chunks = no shores = no hole = no gap.

APPLIED TO COMMUNICATION:

  Sender has chunks A, B, C, D about topic X.
  Sender wants to transfer → speaks/writes/presents about X.

  Receiver does NOT HAVE chunks A, B, C, D.
  → Receiver has no shores → no hole → NO GAP about X.
  → Receiver hears sender speak → but:
    ① No gap → no direction → doesn't know "what does this fill"
    ② No shores → new chunks have nowhere to attach → compile WEAK
    ③ No body-base reward → "heard but immediately forgotten"

  = "SPEAKING" = sender externalizes chunks
  = "UNDERSTANDING" = receiver COMPILES new chunks INTO their own network
  = 2 COMPLETELY DIFFERENT processes

  Sender can externalize perfectly →
  but if receiver has no surrounding chunks →
  compilation DOES NOT OCCUR at any meaningful level.
```

### §1.2 — Why even experts cannot absorb cross-domain dumps

```
🟡 CROSS-DOMAIN DUMP FAILURE:

  A math expert HEARS a list of hormones:
    "Cortisol, oxytocin, serotonin, dopamine, endorphin..."
    → Chunks about neurochemistry: ≈ 0
    → No shores → no gap → labels fly past → compile EXTREMELY WEAK
    → Body experience: "heard the sounds but understood nothing"

  A neuroscience expert HEARS a math proof:
    "For ε > 0, there exists δ > 0 such that..."
    → Chunks about epsilon-delta: ≈ 0
    → No shores → labels fly past
    → SAME experience: "heard the sounds but understood nothing"

  BOTH ARE EXPERTS — but in DIFFERENT DOMAINS.
  → Chunks are DOMAIN-SPECIFIC: math ≠ neuroscience ≠ psychology ≠ engineering
  → Cross-domain dump = sending labels into a network WITH NO SHORES
  → = "Not knowing = no gap" violated at EXPERT level
  → Expertise does NOT automatically transfer cross-domain

  ⭐ CONSEQUENCE:
  "Objective value" of information does NOT determine comprehension.
  E=mc² has enormous value → but presented to someone who doesn't know physics
  → "oh, interesting" but no body-level reward (Gap-Direction.md §3.3).
  = Reward = f(gap direction match), NOT f(information value).
```

### §1.3 — Consequence: must DETECT first, BUILD chunks first

```
🟡 2 CONSEQUENCES FOR ALL COMMUNICATION:

  ① MUST DETECT receiver's chunk landscape FIRST:
     → What chunks receiver has → which gaps EXIST → which directions are ACTIVE
     → SAME content, presented DIFFERENTLY for different chunk landscapes
     → Gap-Direction.md §3.3 table: school student vs university student vs Einstein
       receive SAME E=mc² → COMPLETELY DIFFERENT reward
     → = Must know where receiver IS before speaking

  ② MUST BUILD surrounding chunks BEFORE presenting core:
     → "Not knowing = no gap" → must create shores FIRST
     → Creating shores = build related chunks → gap forms NATURALLY
     → AFTER gap forms + direction is clear → present core → fill gap → reward
     → = Education principle (00-External-Mechanism.md §7):
       "Build the question before giving the answer"

  CONNECTION to existing framework:
  → Gap-Direction.md §3: "not knowing = no gap" (genesis principle)
  → Gap-Direction.md §8: gap install = build chunks → gap emerges
  → 00-External-Mechanism.md §7: Education = batch Direction B
  → 01c-Chunk-Discovery-Lifecycle.md §2: accumulate → vague → curious → clarify → label
  → = This file EXTENDS: HOW to build chunks optimally for cross-network transfer
```

## §2 — ANALOGY MECHANISM

### §2.1 — What analogy IS at the chunk level

```
🟡 ANALOGY = CROSS-DOMAIN CHUNK MAPPING:

  DEFINITION:
    Analogy = using chunks from Domain A (already known) to CREATE TEMPORARY SHORES
    for gaps in Domain B (not yet known).

  MECHANISM:
    ① Receiver ALREADY HAS rich chunks about Domain A
    ② Sender SAYS: "Domain B is LIKE Domain A in that..."
    ③ Receiver's brain: Domain A chunks FIRE
    ④ Activation from Domain A creates TEMPORARY SHORES around topic Domain B
    ⑤ Temporary shores → gap forms → direction points toward Domain A
    ⑥ Receiver "understands" — but understands THROUGH THE LENS of Domain A

  EXAMPLE:
    Domain B = PFC (prefrontal cortex) — receiver knows nothing about it
    Domain A = "company CEO" — receiver has rich chunks about it
    Sender: "PFC is like a CEO — makes plans, issues decisions"
    → Receiver's [CEO] chunks fire → creates temporary shores for [PFC]
    → Gap forms: "PFC = CEO" → direction toward Domain A
    → Receiver "understands" — but understands PFC THROUGH the CEO chunks

  ⭐ KEY DISTINCTION:
    Temporary shores = BORROWED chunks, not REAL chunks about Domain B.
    Receiver does not yet have real chunks about PFC.
    Receiver has chunks about "CEO" → used as SUBSTITUTE.
    = F3 install uses BORROWED chunks as scaffolding.
```

### §2.2 — WHY analogy works (when it works)

```
🟡 ANALOGY BYPASSES "NOT KNOWING = NO GAP":

  PROBLEM: Receiver has no chunks in Domain B → no gap → cannot receive.
  BYPASS: Use Domain A chunks (ALREADY PRESENT) → create temporary shores → gap direction forms.

  WHY BYPASS IS POSSIBLE:
    → Chunk activation = spreading activation (01b-Chunk-Activation-Dynamics.md §2)
    → When saying "PFC is like a CEO" → chunks [CEO] fire
    → [CEO] links to [make plans], [organize], [authority]
    → Entire cluster fires → CREATES CONTEXT for "PFC"
    → PFC label ATTACHES to this context (Direction B: label → experience)
    → = Surrogate chunks create shores → gap direction toward "CEO" cluster

  WHEN IT WORKS:
    ① Receiver HAS rich chunks about Domain A
       → Temporary shores THICK → gap direction CLEAR → "understands" quickly
    ② Domain A and Domain B share BASIC STRUCTURE
       → Temporary shores approximately CORRECT → direction approximately matches reality
       → EXAMPLE: "chunk network ≈ graph data structure" for engineer
         = Nodes (chunks), edges (links), traversal (activation spreading)
         = Structural similarity HIGH → analogy TRUSTWORTHY

  🟢 EVIDENCE BASE:
    → Structure-mapping theory (Gentner 1983): analogy works via
      structural alignment between domains
    → Analogical reasoning uses relational matching, not surface features
      (Gentner & Markman 1997)
    → Compound transfer: source domain depth × structural overlap
      = analogy effectiveness (🟡 framework extension)
```

### §2.3 — Analogy quality = f(structural_similarity × receiver_depth)

```
🟡 2 FACTORS DETERMINING ANALOGY QUALITY:

  ① STRUCTURAL SIMILARITY:
     = HOW SIMILAR the structure of Domain A and Domain B are

     HIGH:
       "Chunk network ≈ graph data structure"
       Nodes = chunks ✅, Edges = links ✅, Traversal = activation ✅
       Weight = probability ✅, Properties = attributes ✅
       → Structural similarity ~80% → analogy STRONG

     LOW:
       "PFC ≈ CEO"
       Makes plans ✅, but:
       CEO controls everything ❌ (PFC reach limited ~5%)
       CEO knows best ❌ (PFC does NOT access 95% of processing)
       CEO has absolute authority ❌ (body signals CAN override)
       → Structural similarity ~30% → analogy DANGEROUS

  ② RECEIVER'S DOMAIN A DEPTH:
     = HOW DEEP the receiver's Domain A chunks are

     DEEP (many chunks, many links, much experience):
       → Temporary shores THICK + DETAILED → gap direction CLEAR
       → EXAMPLE: engineer who knows graph theory DEEPLY → chunk network analogy
         gives EXTREMELY CLEAR direction → "understands" quickly + accurately

     SHALLOW (only label, little experience):
       → Temporary shores THIN → gap direction VAGUE
       → EXAMPLE: hearing "graph" for the first time → analogy DOES NOT HELP → 2 new things at once

  ⭐ FORMAL:
    Analogy_Quality = f(Structural_Similarity × Receiver_Domain_A_Depth)
    → Both low → analogy FAILS
    → Similarity high + depth high → analogy BEST
    → Similarity low + depth high → analogy MOST DANGEROUS
      (receiver "understands" incorrectly AND IS CONFIDENT → hardest to correct)
```

## §3 — 4 RISKS OF ANALOGY

### §3.0 — Overview

```
⭐ THESE 4 RISKS = inevitable consequences of the §2 mechanism:

  Analogy uses BORROWED chunks → TEMPORARY shores → gap direction POINTS TOWARD Domain A.
  Real Domain B has DIFFERENT structure from Domain A.
  → 4 risks emerge from this difference:

  ┌──────────────────────────────────────────────────────────────────────┐
  │  Risk 1: Gap Direction Mismatch — direction from Domain A ≠ real    │
  │  Risk 2: Shift Cost — shifting from analogy → real = competitive    │
  │          re-linking, potentially HARDER than learning from scratch  │
  │  Risk 3: False Specificity — wrong constraints from Domain A        │
  │  Risk 4: Double Unknown — receiver also doesn't know Domain A       │
  └──────────────────────────────────────────────────────────────────────┘
```

### §3.1 — Risk 1: Gap Direction Mismatch

```
⭐ TEMPORARY SHORES FROM DOMAIN A → GAP DIRECTION POINTS TOWARD DOMAIN A:

  MECHANISM:
    Gap direction = f(surrounding chunk network structure)
    (Gap-Direction.md §1.1)

    Surrounding chunks = BORROWED from Domain A
    → Gap direction points toward Domain A, NOT Domain B
    → When encountering REAL mechanism (Domain B) → direction DOES NOT MATCH

  EXAMPLE — "PFC = CEO":
    Analogy installs:
      [PFC] → [control] + [knows best] + [issues orders → obeys]

    Real mechanism (Core-Software.md §6.1):
      [PFC] → [reach limited ~5%] + [does NOT access body processing]
             + [only HOLDS DIRECTION, does not control details]

    → Gap direction analogy: "PFC controls everything, knows all"
    → Gap direction real: "PFC observes results, holds direction, body executes"
    → 2 directions are NEARLY OPPOSITE

    → When receiver LATER encounters the real mechanism:
      "Consciousness only perceives ~5% of processing" → CONFLICTS with "CEO knows all"
      → ACC detects inconsistency (Bush et al. 2000) → dissonance
      → This dissonance was UNNECESSARY — created BY the analogy

  FORMAL:
    Gap_Direction_analogy ≠ Gap_Direction_real
    → Mismatch cost when upgrading from analogy to real understanding
    → Mismatch = PROPORTIONAL to structural dissimilarity
    → High dissimilarity = high mismatch = high correction cost
```

### §3.2 — Risk 2: Shift Cost (Competitive Re-linking)

```
⭐ SHIFTING FROM ANALOGY → REAL = COMPETITIVE RE-LINKING:

  MECHANISM (01b-Chunk-Activation-Dynamics.md §3):
    Old links CANNOT be deleted — they only weaken as new links COMPETE.

    When analogy has compiled:
      [PFC] → [CEO] → [control, command, knows all]
      = Links ALREADY FORMED, possibly repeated many times

    When learning the real mechanism:
      [PFC] → [observer] → [reach limited, holds direction, body executes]
      = NEW links must COMPETE with old links

    Shift = must SIMULTANEOUSLY:
      ⓐ Build new links (PFC → observer → reach limited...)
      ⓑ Old links STILL fire (PFC → CEO → control...)
      ⓒ 2 sets of links COMPETE for activation probability (01b §3.1)
      ⓓ Retrieval-induced forgetting: using new link SUPPRESSES old link temporarily
         but old link STILL EXISTS, can re-fire under stress
         (🟢 Anderson et al. 1994)

  EXAMPLE — "PFC = CEO" → "PFC = observer":
    Chunks surrounding "CEO":
      [affection (if "mother" was used)], [strategy], [organization], [positional authority]
    Chunks surrounding "observer":
      [perceives], [limited], [guides], [body executes on its own]
    = 2 COMPLETELY DIFFERENT chunk sets
    → Competitive re-linking = energy cost each time the wrong association fires

  ⭐ KEY INSIGHT:
    Shift from analogy → real CAN BE HARDER than learning real from the start.

    Learning from scratch: only needs to BUILD new links (1 task)
    Shift: must BUILD new + COMPETE with old + FILTER right/wrong (3 simultaneous tasks)

    → If analogy is used as a STEPPING STONE (will upgrade later)
      → shift cost may > cost of building from scratch
    → = Analogy "shortcut" upfront → PAY MORE later
```

### §3.3 — Risk 3: False Specificity

```
⭐ ANALOGY INSTALLS WRONG CONSTRAINTS FROM DOMAIN A:

  MECHANISM:
    Gap-Direction.md §4.2: Specificity = constraints narrow or broad
    Analogy provides specificity from Domain A.
    These constraints MAY BE WRONG for Domain B.

  EXAMPLE — "PFC = CEO":
    Domain A constraint: "CEO issues orders → employees obey"
    Installed in Domain B: "PFC issues orders → body obeys"

    Real mechanism (Core-Software.md §6.1):
      The unconscious does NOT obey PFC commands.
      You do not decide your heart rate, do not choose when you get hungry.
      PFC holds direction — body decides the details on its own.

    → Constraint "issues orders → obeys" = FALSE SPECIFICITY
    → Receiver believes "if I COMMIT myself, PFC will be able to control it"
    → When it fails (willpower cannot overcome the urge) → "I am weak"
    → Unnecessary dissonance — created by false constraint from analogy

  FORMAL:
    Specificity_analogy installs WRONG constraints
    → Wrong constraints → wrong predictions → wrong expectations
    → Wrong expectations → dissonance when encountering reality
    → = Analogy creates harmful expectations ABOUT Domain B

  SEVERITY:
    → Surface constraints wrong → mild confusion (easy to correct)
    → Deep structural constraints wrong → fundamental misunderstanding (hard to correct)
    → EXAMPLE "CEO" = deep structural constraint → HARD TO CORRECT
      (affects understanding of the entire conscious-unconscious relationship)
```

### §3.4 — Risk 4: Double Unknown

```
🟡 ANALOGY ONLY WORKS WHEN RECEIVER HAS DOMAIN A CHUNKS:

  MECHANISM:
    Analogy bypasses "not knowing = no gap" via SURROGATE chunks.
    IF receiver ALSO doesn't know Domain A → NO surrogate chunks.
    → 2 new things at once: Domain A + Domain B.

  EXAMPLE — "PFC = CEO" for a 5-year-old:
    The child doesn't know what a CEO DOES.
    Chunks [CEO]: ≈ 0 or very shallow.
    → Analogy = 2 unknowns: "PFC" + "CEO"
    → Body must simulate BOTH at the same time
    → WM capacity ~3-5 items (01b §2) → 2 unknowns ≈ at capacity
    → Result: confused, DOES NOT HELP at all

  EXAMPLE — Technical analogy for non-technical audience:
    "Chunk network ≈ graph data structure"
    → If receiver doesn't know graph theory:
      "graph data structure" = unknown #2
      → Analogy ADDS burden instead of reducing it
    → If receiver knows graph theory DEEPLY:
      → Analogy EXCELLENT (rich surrogate chunks)
    → = SAME analogy: excellent for 1 audience, harmful for another

  ⭐ CONSEQUENCE:
    Analogy effectiveness = RECEIVER-DEPENDENT.
    Sender CANNOT choose the "best" analogy WITHOUT knowing
    whether receiver has Domain A chunks.
    → = Must DETECT receiver's chunk landscape FIRST (§7).
```

## §4 — WHEN TO USE ANALOGY vs NOT

### §4.1 — Criterion 1: Structural similarity

```
⭐ USE WHEN DOMAIN A AND DOMAIN B SHARE STRUCTURE:

  ✅ SHOULD USE:
    Domain A and Domain B have HIGH structural overlap.
    = Gentner 1983: relational similarity, not just surface.

    Good examples:
    → "Chunk network ≈ graph" for engineer
      Nodes=chunks ✅, Edges=links ✅, Weight=probability ✅
      Traversal=spreading activation ✅
      → Structural overlap ~80%

    → "Gap = hole in puzzle" (Gap-Direction.md §2.2)
      Shape=direction ✅, Surrounding pieces=chunks ✅
      → Structural overlap ~60% (puzzle static, chunks dynamic — marked)

  ⚠️ CAUTION:
    Surface similar but DIFFERENT structure.
    → "PFC = CEO"
      Both "make decisions" (surface ✅)
      But: scope, authority, mechanism DIFFERENT (~30% structural overlap)
      → Surface misleading

  ❌ DO NOT USE:
    Structure FUNDAMENTALLY DIFFERENT.
    → Any analogy with structural overlap < 40%
    → Will install MORE wrong associations than right ones
```

### §4.2 — Criterion 2: Endpoint vs stepping stone

```
⭐ ANALOGY AS ENDPOINT VS STEPPING STONE:

  ✅ SHOULD USE WHEN ENDPOINT:
    Receiver does NOT NEED to go deeper.
    Analogy = "good enough" for receiver's purpose.
    → No shift cost because there IS NO upgrade.

    EXAMPLE: "Neurons firing like wires transmitting signals"
    → For non-scientist who wants basic understanding → SUFFICIENT
    → No correction needed because deeper level NOT NEEDED

  ⚠️ DANGEROUS WHEN STEPPING STONE:
    Receiver WILL NEED to upgrade later.
    → Analogy creates links → upgrade = competitive re-linking (§3.2)
    → Shift cost = additional effort + UNNECESSARY dissonance
    → May be HARDER than building correctly from the start

    EXAMPLE: "PFC = CEO" → then learns for real
    → Links [PFC→CEO→control] COMPETE with
      [PFC→observer→reach limited]
    → Receiver must simultaneously: break old + build new + filter right/wrong
    → = Stepping stone MORE COSTLY than no stepping stone
```

### §4.3 — Criterion 3: Marked approximate vs identity

```
🟡 HOW ANALOGY IS PRESENTED DETERMINES RISK LEVEL:

  ✅ SAFER — "like... but different in that..."
    = Clearly MARKS the BOUNDARY between Domain A and Domain B

    EXAMPLE: "PFC is IN SOME WAYS LIKE a CEO in making plans,
         BUT DIFFERENT in that: PFC cannot control the body,
         only perceiving about 5% of processing."

    → Receiver knows: analogy = approximation, has limits
    → Chunks compile WITH boundary tag: [similar but different]
    → When encountering real mechanism → LESS dissonance (already expected differences)

  ❌ DANGEROUS — "PFC IS the CEO"
    = Presented as IDENTITY, no boundary marked

    → Receiver compiles: [PFC = CEO] (identity link, STRONG)
    → When encountering real mechanism → MAXIMUM dissonance
      ("but I was told PFC is the CEO?")
    → Identity link HARDER to shift than similarity link
      (01b §3: link strength ∝ compile strength)
```

### §4.4 — Criterion 4: Whether receiver has Domain A chunks

```
🟡 ALWAYS CHECK: WHETHER RECEIVER HAS DOMAIN A CHUNKS:

  YES → analogy works (surrogate chunks available)
  NO → Double Unknown → analogy harmful (§3.4)

  HOW TO CHECK:
  → Ask about experience: "Do you know graph theory?"
  → Observe vocabulary: if using technical terms → HAS chunks
  → Observe response: confused face → DOES NOT HAVE chunks

  ⭐ IF UNSURE → use experience-based language instead (§5)
    = Safer default — uses chunks EVERYONE ALREADY HAS
    = Slower but DOES NOT CREATE wrong associations
```

### §4.5 — Decision matrix

```
🟡 DECISION TABLE:

  ┌─────────────────────┬────────────────────┬──────────────┬─────────────────────┐
  │                     │ Struct. sim.       │ Endpoint?    │ Knows Domain A      │
  │                     │ high (>60%)        │              │ deeply?             │
  ├─────────────────────┼────────────────────┼──────────────┼─────────────────────┤
  │ ✅ Use analogy      │ ✅                 │ ✅           │ ✅                  │
  │ ⚠️ Use + mark clearly│ ✅                │ ❌ (stepping)│ ✅                  │
  │ ⚠️ Proceed with caut│ ❌ (low)           │ ✅           │ ✅                  │
  │ ❌ Do NOT use       │ ❌                 │ ❌           │ any                 │
  │ ❌ Do NOT use       │ any                │ any          │ ❌                  │
  └─────────────────────┴────────────────────┴──────────────┴─────────────────────┘

  DEFAULT: When in doubt → use experience-based language (§5).
```

  → Everyone ALREADY HAS these chunks → REAL shores (not borrowed)
  → Gap direction points CORRECTLY toward the real mechanism:
    "Ah, consciousness does not control everything — it observes"
  → Does NOT create wrong associations → NO shift cost needed later
```

### §5.2 — Why experience-based language is better than analogy

```
🟡 COMPARISON AT THE CHUNK LEVEL:

  ANALOGY:
    Shores = BORROWED chunks (Domain A)
    → Gap direction points toward Domain A
    → When upgrading → competitive re-linking (§3.2)
    → FORMAL: Gap_Direction_analogy ≈ Gap_Direction_Domain_A ≠ Gap_Direction_real
    → = Fast install, potential wrong direction, shift cost

  EXPERIENCE-BASED:
    Shores = EXISTING chunks (receiver's own observations)
    → Gap direction points toward the REAL mechanism
    → When going deeper → BUILDS ON existing, no shift needed
    → FORMAL: Gap_Direction_experience ≈ Gap_Direction_real
    → = Slower install, correct direction, no shift cost

  WHY REAL SHORES ARE BETTER THAN TEMPORARY SHORES:
    Real shores = chunks receiver ALREADY HAS from their own experience
    → Links already strong (compiled over years of experience)
    → Direction CORRECT because shores BELONG TO the real domain
    → No false constraints from another Domain
    → When gap fills → genuine BODY-BASE reward (gap direction match)

  EXAMPLE:
    ANALOGY: "PFC = CEO → gives orders to the body"
    → Direction: [control, command, knows all]
    → WRONG → shift cost later

    EXPERIENCE-BASED: "You do not decide your heart rate.
    You do not choose when you get hungry.
    Consciousness is the latecomer."
    → Direction: [limited, observes, receives results]
    → CORRECT → build on later, no shift cost
```

### §5.3 — Limitation: experience-based language is slower

```
🟡 TRADEOFF: ACCURACY vs SPEED

  ANALOGY:
    → "Shortcut" — fast BUT risky
    → 1 sentence can "unlock" understanding (if analogy is good)
    → BUT: the understanding CAN have wrong direction

  EXPERIENCE-BASED:
    → Accurate BUT requires more building
    → Must build each observation chunk → gap gradually forms
    → Receiver "understands" SLOWER but MORE CORRECTLY
    → NO shift cost → NET faster for long-term understanding

  WHEN SPEED IS MORE IMPORTANT:
    → Emergency situation (need to understand now, no time to build)
    → Endpoint understanding (no need to go deeper)
    → Quick orientation (will learn properly later, just need overview)
    → = Analogy OK for these contexts (IF structural similarity is high)

  WHEN ACCURACY IS MORE IMPORTANT:
    → Education (student will build ON TOP of this understanding)
    → Therapy (wrong understanding → harmful expectations)
    → Self-understanding (wrong gap direction → wrong self-model)
    → = Experience-based language PREFERRED for these contexts
```

## §6 — MINI-GAP CHAIN THEORY

### §6.1 — Principle: each segment = fill 1 small gap → create curiosity for the next gap

```
⭐⭐ MINI-GAP CHAIN = APPLIED OSCILLATION DYNAMICS:

  Gap-Direction.md §7.5 oscillation:
    fill₁ → reward₁ + new chunks₁ → detect new gap₂
    fill₂ → reward₂ + new chunks₂ → detect new gap₃
    ...
    fillₙ → BREAKTHROUGH: many chunks suddenly UNIFY

  APPLIED TO COMMUNICATION:
    Each presentation segment = 1 mini-arc:
      ① START from chunks the receiver ALREADY HAS (existing observation)
      ② Create a small CONFLICT or highlight an UNNAMED PATTERN
      ③ Gap direction forms: "ah, something is here"
      ④ FILL gap: present 1 insight — just enough, NOT too much
      ⑤ Fill → reward (body recognizes: "ah, that's right!")
      ⑥ New chunks compile → detect NEW inconsistency → NEW gap
      ⑦ New gap = CURIOSITY for next segment

    = Each segment is 1 oscillation cycle
    = Chain of segments = CHAIN of oscillation cycles
    = Reader/listener PULLED FORWARD by own curiosity

  EXAMPLE — Chain for "why am I lazy?":
    Step 1: "Have you ever been too lazy to eat when you were hungry?"
    → Receiver: "No, when I'm hungry I just eat." → chunks fire
    → Gap: "So why be lazy about other things?"

    Step 2: "The body is not lazy — it simply hasn't seen ENOUGH REASON to act."
    → Fill gap: "Ah, not lazy — just insufficient reason"
    → New gap: "So what does sufficient reason mean?"

    Step 3: "When hungry, the body sees CLEARLY: eat → no more hunger → feels good.
    But go to the gym — the body hasn't seen a clear reward yet."
    → Fill gap: "Ah, reward must be CLEAR TO THE BODY, not just to the mind"
    → New gap: "So how do we make the reward clear to the body?"

    → Each step: fill + curiosity → pull forward → VOLUNTARY deepening
```

### §6.2 — Design: gap direction of each segment must be CORRECTLY ORIENTED

```
🟡 MINI-GAP CHAIN DESIGN PRINCIPLES:

  ① DIRECTION CONTROL:
     Each fill → new chunks compile → new shores → new gap.
     New gap must point TOWARD real mechanism (not away).

     CORRECT DIRECTION:
       Step 1: "you're not lazy when hungry" → direction: "what's different?"
       Step 2: "insufficient reason" → direction: "what is reason?"
       Step 3: "body needs clear reward" → direction: "how to create reward?"
       → Each step: direction LEADS TOWARD the real mechanism (body-base reward)

     WRONG DIRECTION (if wrong analogy used):
       Step 1: "PFC = CEO" → direction: "CEO controls"
       Step 2: "CEO knows best" → direction: "willpower decides"
       Step 3: "try harder" → direction: AWAY from the real mechanism
       → Each step: direction MOVES AWAY from mechanism → harder to correct later

  ② EACH STEP BUILDS ON PREVIOUS:
     Step N chunks = SHORES for Step N+1 gap.
     → If step N is wrong → step N+1 shores wrong → direction wrong → COMPOUND error
     → = Why quality of EACH step matters

  ③ VERIFY VIA RECEIVER'S RESPONSE:
     After each step: check whether receiver understood CORRECTLY
     → "Is that what you understood?" or observe response
     → If direction is off → correct BEFORE building further
     → = Real-time direction monitoring
```

### §6.3 — Pacing: just enough → reward → curiosity → continue

```
🟡 GOLDILOCKS PER STEP:

  TOO MUCH AT ONCE:
    → WM capacity ~3-5 items (01b-Chunk-Activation-Dynamics.md §2)
    → Dump 10 concepts at once → overwhelm → "too much, can't understand"
    → Body: cortisol rises → threat tag → "this topic is too hard, don't like it"
    → = BAD EXPERIENCE compiled → harder to re-engage

  TOO LITTLE:
    → Each step only 1 trivial observation → gap direction does NOT form
    → "Losing interest" — gap too small → reward too small → curiosity fades
    → = Energy drops quickly → "talked forever but got nowhere"

  JUST RIGHT (Goldilocks):
    → Enough MISMATCH to create gap (curiosity trigger)
    → Enough FILL to provide reward (body: "ah, that's right!")
    → Enough NEW CHUNKS for next gap to form
    → = Reward SUSTAINS engagement → chain CONTINUES

  PACING FACTORS:
    → Receiver's existing chunks: many → can pace faster
    → Emotional state: stressed → slower pace (WM reduced)
    → Complexity of topic: complex → smaller steps
    → = Pace = ADAPTIVE, not fixed
    → Connection to 00-External-Mechanism.md §7: Education failure mode ①
      "Label without experience = SHALLOW chunk"
      → Too fast = too many labels → shallow compile
```

### §6.4 — Entry point: START FROM observations the receiver ALREADY HAS

```
⭐ "NOT KNOWING = NO GAP" → MUST START FROM EXISTING CHUNKS:

  ENTRY POINT = chunks the receiver ALREADY HAS.
  If starting from chunks receiver does NOT YET HAVE → "not knowing = no gap"
  → no engagement → chain cannot begin.

  PROFILE-DEPENDENT ENTRY POINTS:

  General audience (observation-dominant chunks):
    → Start from everyday behaviors: eating, sleeping, stress, laziness, sadness
    → EXAMPLE: "Have you ever been hungry but not wanted to eat?"
    → Chunks [hunger], [want to eat] = universal → REAL shores → gap forms
    → = experience-based language (§5): observer-level, everyone has these

  Psychology professional (domain chunks + behavioral chains):
    → Start from existing professional observations
    → EXAMPLE: "Do you find that every client describes knowing but being unable to act?"
    → Chunks [client behavior], [know-do gap] = domain-specific → DEEP shores
    → Can be more direct, less building needed

  Engineer / technical professional (formal structure chunks):
    → Start from structural patterns they are familiar with
    → EXAMPLE: "Every system has a feedback loop — the body is no different."
    → Chunks [feedback loop], [system] = technical → STRUCTURAL shores
    → Can use structural analogy (IF structural similarity is high, §4.1)
    → Build psychology bottom-up from structural foundation
    → Risk: engineer may over-systematize (body ≠ deterministic system)
    → Pacing: can go faster on structure, slower on body/feeling territory

  → = Entry point = receiver-dependent. DETECT FIRST (§7).
```

### §6.5 — Progression: observation → pattern → mechanism

```
🟡 PROGRESSIVE DEEPENING VIA MINI-GAP CHAIN:

  CONNECTION to Feeling-Literacy-Training.md §3:
    Stage 1: Body Awareness → detect raw signals
    Stage 2: Signal Discrimination → which channel, what intensity
    Stage 3: Pattern Recognition → this feeling = this situation
    Stage 4: Label Attachment → name → logic-planning access
    Stage 5: Meta-Feeling → recursive awareness + calibration
    → Each stage = deeper chunks → richer gap landscape

  APPLIED TO KNOWLEDGE TRANSFER:

  LEVEL 1 — OBSERVATION:
    "Do you recognize X in your own life?"
    → Receiver recognizes own experience → chunks fire → "yes, that's right!"
    → = Build shores from existing observations → gap direction forms
    → experience-based language (§5)

  LEVEL 2 — PATTERN:
    "When does X happen for you? Is there a pattern?"
    → Receiver detects pattern across observations → new insight
    → = Gap fill: "ah, it always happens when Y" → reward
    → Mini-gap: "why is Y connected?"

  LEVEL 3 — MECHANISM (if receiver wants to go deeper):
    "Pattern X-Y occurs BECAUSE of mechanism Z."
    → Now receiver HAS chunks (observation + pattern) → REAL shores
    → Z = fills REAL gap (not surrogate) → body-level reward
    → = Core-Software.md language (mechanism, deeper)

  ⭐ RECEIVER CHOOSES DEPTH:
    Not everyone wants to go to Level 3.
    Level 1 = SUFFICIENT for basic self-awareness.
    Level 2 = SUFFICIENT for behavior change.
    Level 3 = for researcher, therapist, educator.
    → "Want to know more?" = respect body-base pacing.
    → = Receiver's gap direction DECIDES depth, not sender.
```

## §7 — AUDIENCE CHUNK DETECTION

### §7.1 — WHY you must detect before presenting

```
⭐ SAME CONTENT, DIFFERENT CHUNK LANDSCAPES → MUST PRESENT DIFFERENTLY:

  Gap-Direction.md §3.3 table:
    School student: minimal physics chunks → E=mc² → "oh, interesting" (no body reward)

    Einstein: physics chunks YEARS deep → E=mc² → "intensely pleasant"
    → SAME information → COMPLETELY DIFFERENT reward
    → BECAUSE gap landscape DIFFERENT → gap direction DIFFERENT → match quality DIFFERENT

  CONSEQUENCE FOR COMMUNICATION:
    → Sender MUST know receiver's chunk landscape FIRST
    → Only then can choose: entry point, pacing, language, analogy vs experience-based
    → NOT detecting → risk: wrong entry point → no engagement
      or wrong analogy → harmful associations

  WHY "LENS PER AUDIENCE" IS NOT THE SOLUTION:
    → Pre-configured lens (Ask-AI-Parent.md, Ask-AI-Leader.md) = static category
    → Each lens = pre-installs gap direction INTO the sender (AI)
    → Parent asks "I'm so stressed" → AI biases answer toward parenting stress
    → But stress might be from work, relationship, health
    → = Lens installs BIAS, not detection
    → = F3 External Install of bias INTO the communication tool itself
    → Framework principle: each person = unique chunk landscape,
      CANNOT be categorized into 1 lens
```

### §7.2 — WHAT TO DETECT

```
🟡 4 SIGNALS TO ESTIMATE CHUNK LANDSCAPE:

  ① PROFESSION → domain chunks estimate:
     → Engineer → formal structure chunks likely present
     → Teacher → behavioral observation chunks likely present
     → Doctor → body/health chunks likely present
     → Student → depends on field + depth
     → = Profession = ROUGH estimate, NOT deterministic

  ② VOCABULARY USED → label depth estimate:
     → Uses "stress" vs "cortisol" vs "hypothalamic-pituitary-adrenal axis"

     → Technical label ("cortisol") → Direction B installed, depth unknown

     → "Why does dopamine signaling decrease?" → gap about neuroscience (expert)
     → = Question STRUCTURE reveals gap DIRECTION
     → = Most direct signal: "questions reveal what they DO NOT YET know
       but ALREADY HAVE sufficient shores to ask"

  ④ EMOTIONAL CONTEXT → body-state affects presentation pacing:
     → Calm curiosity → can handle faster pace
     → Distressed → slower pace, more validation first
     → Defensive → even slower, establish safety first
     → = Emotional state = pacing modifier, not content modifier

## §8 — 2-FORCE TENSION AND RESOLUTION

### §8.1 — Force 1: Match user chunks (comprehension + reward)

```
🟡 FORCE 1 — MATCH EXISTING CHUNKS:

  Receiver ENGAGES when content MATCHES existing chunk landscape.
  → Match → spreading activation → recognition → "ah, I know this"
  → Recognition = reward (opioid micro-dose)
  → Engagement SUSTAINS → conversation continues

  IF NO match:
  → "Not knowing = no gap" → content flies past
  → No engagement → conversation dies
  → = "Great book but can't get into it"

  FORCE 1 = PULL toward receiver's existing chunks.
  = Use THEIR language, THEIR examples, THEIR experience.
  = "Start from where THEY are, not where YOU are."
```

### §8.2 — Force 2: Present accurately (real understanding)

```
🟡 FORCE 2 — PRESENT MECHANISM CORRECTLY:

  Framework mechanism = WHAT sender wants receiver to understand.
  → Mechanism is what it IS — CANNOT be changed to fit receiver
  → EXAMPLE: PFC reach limited ~5% = fact → CANNOT say "PFC controls everything"
    just because receiver WANTS to hear that

  IF accuracy is sacrificed for match:
  → Wrong direction analogy → receiver "understands" INCORRECTLY
  → Wrong understanding compiled → shift cost later (§3.2)
  → = "Understands but understands incorrectly" → worse than "doesn't understand yet"

  FORCE 2 = PULL toward accurate mechanism.
  = Present WHAT IS, not what receiver expects.
  = "Don't tell them what they want to hear — tell them what's real."
```

### §8.3 — Connection to 2-Layer Model

```
🟡 GAP-DIRECTION.MD §6: 2-LAYER MODEL APPLIED:

  Layer 1 (Signal mechanism): must TRIGGER interest
    → = Force 1: content must match enough for signal to fire
    → If NO match → no signal → no attention → communication fails

  Layer 2 (Direction content): gap direction must be CORRECT
    → = Force 2: gap formed must point toward REAL mechanism
    → If direction WRONG → "wrong understanding" → shift cost later

  → NEED BOTH:
    Layer 1 fires (engagement) + Layer 2 correct (accuracy)
    = Comprehension + Correct understanding
```

### §8.4 — Resolution: mini-gap chain from Force 1 → Force 2

```
⭐⭐ RESOLUTION — DO NOT CHOOSE ONE, USE BOTH IN SEQUENCE:

  START from Force 1:
    → Match existing chunks (signal fires, interest triggered)
    → Use receiver's language, receiver's observations
    → = Entry point = receiver's territory

  PROGRESS toward Force 2:
    → Each mini-gap fill → new chunks compile
    → New chunks = new shores → gap direction SHIFTS toward mechanism
    → Each step: slightly more accurate, slightly less "familiar"
    → Receiver PULLED FORWARD by own curiosity (§6.1)

  END at Force 2 (or depth receiver chooses):
    → If receiver continues → reach mechanism level
    → If receiver stops → understanding AT THEIR DEPTH
    → Both valid — receiver's gap direction DECIDES

  = MINI-GAP CHAIN IS THE RESOLUTION:
    Chain START = Force 1 territory (receiver's chunks)
    Chain END = Force 2 territory (real mechanism)
    Chain PROGRESSION = gradual direction shift via mini fills
    → = No tradeoff needed — SEQUENTIAL integration

  EXAMPLE:
    Force 1 start: "You're never too lazy to eat when you're hungry, are you?"
    → Match: receiver RECOGNIZES own experience → engaged

    Progress: "Why? Because the body sees CLEARLY: eat → hunger gone → feels good."
    → Slightly new: "the body sees clearly" → direction toward body mechanism

    Progress: "But go to the gym — the body hasn't seen a clear reward yet."
    → Newer: "clear reward" = mechanism concept building

    End (if they want depth): "The body needs 3 things to perceive it as worthwhile:
    reward close enough, clear enough, and large enough relative to effort."
    → = Mechanism level, arrived via receiver's own curiosity
```

### §8.5 — When 2 forces CONFLICT irreconcilably

```
🟡 WHEN RECEIVER CHUNKS ARE TOO FAR FROM THE MECHANISM:

  Sometimes: receiver's chunk landscape is SO DIFFERENT from the mechanism
  → Mini-gap chain CANNOT bridge the gap in 1 conversation
  → EXAMPLE: receiver deeply believes "willpower decides everything"
    + framework: "willpower = ~5% PFC, body decides mostly"
    → Gap too large → too many steps needed

  SOLUTION:
    ① Accept partial understanding — depth receiver CAN reach in 1 session
    ② Invite deeper exploration: "Want to know more?"
    ③ = Respect body-base pacing: each fill needs consolidation (sleep, more lived experience)
    ④ Next conversation = start from NEW chunks compiled since last time
    → = Multiple sessions > 1 marathon session

  DO NOT force:
    → Forcing deep understanding when chunks are insufficient = violates "not knowing = no gap"
    → Receiver "understands" via label (Direction B) but body does NOT confirm
    → = "False explicit" (Somatic-Articulation-Loop.md §1.3)
    → = Verbal knowledge exists, felt sense missing
    → HARMFUL: receiver thinks they understand but doesn't
```
